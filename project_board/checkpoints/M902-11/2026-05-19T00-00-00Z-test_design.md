## Test Structure and Methodology

### Mocking Strategy

All tests use `unittest.mock.patch` to mock tool execution (`_run_import_linter`, `_run_eslint`, `_run_semgrep`, `_run_jscpd`, `_run_radon`). This approach:
- Isolates gate logic from tool implementations (no external dependencies required)
- Allows deterministic test vectors with known inputs/outputs
- Supports testing error paths (timeouts, tool failures, missing tools)
- Enables fast test execution (no subprocess overhead)

Mocking is appropriate here because:
1. Tools are external systems (import-linter, eslint, semgrep, jscpd, radon)
2. Gate's responsibility is orchestration and result aggregation (not tool functionality)
3. Tool correctness is verified separately (tool vendor responsibility)

### Test Organization

Tests organized by requirement for traceability:
- **Requirement 01 (Gate Module and Registry):** Validates file existence, importability, registry entries, run() function contract
- **Requirement 02 (Output Contract):** Validates output schema, field types, status determination, severity sorting, timestamp format
- **Requirement 03 (Tool Orchestration):** Validates all tools invoked, timeouts, unavailability handling, deduplication
- **Requirements 04–08 (SRP, Circulars, Duplication, Complexity, Async):** Direct test vectors from spec (TV-01–TV-20)
- **Requirement 11 (Error Handling):** Validates no silent exceptions, timeout/error/unavailable tool handling
- **Requirement 12 (NFR):** Validates deterministic execution (TV-32), exit codes (integration-level)
- **Edge Cases:** Empty codebase (TV-29), all tools unavailable (TV-30), JSON serializability, mode parameter

## Key Testing Decisions (Checkpoint Protocol)

### 1. Mock Internal Tool Functions, Not Gate Module Itself

**Would have asked:** Should tests mock `subprocess.run` directly or mock gate's internal `_run_*` functions?

**Assumption made:** Mock gate's internal `_run_import_linter()`, `_run_eslint()`, etc. functions. This:
- Reduces test brittleness (internal implementation details don't change contract)
- Simplifies assertion setup (return violation dicts directly)
- Isolates tool integration from gate logic

**Confidence:** High — aligns with spec's requirement to test gate behavior, not tool behavior.

---

### 2. Test Tool Failures Without Subprocess Infrastructure

**Would have asked:** Should tests use real subprocess calls or mock subprocess?

**Assumption made:** Mock tool execution functions, not subprocess. When testing timeout/error handling:
- Raise `subprocess.TimeoutExpired` from mock `_run_semgrep()`
- Raise `FileNotFoundError` from mock `_run_import_linter()`
- Raise `json.JSONDecodeError` from mock `_run_semgrep()` (to simulate parse error)

**Confidence:** High — tests can't reliably trigger real subprocess timeouts; mocking is standard for external system resilience testing.

---

### 3. Status Determination Logic

**Would have asked:** How should status be computed? Should tests prescribe the exact logic?

**Assumption made:** Spec requires:
- `PASS` if no violations or shadow mode
- `FAIL` if any ERROR-severity violations in blocking mode
- `ESCALATE` if CRITICAL violations or architecture_score <= 30

Tests verify status is set correctly for each severity level, but do NOT prescribe the exact threshold computation (deferred to implementation).

**Confidence:** Medium — spec is somewhat vague on "weighted average" risk score computation. Implementation will define thresholds; tests validate they are consistent.

---

### 4. Requirement 09 (Observability) and Requirement 10 (Mutation) Not Directly Tested

**Would have asked:** Should we implement tests for OB-01, OB-02, OB-03, MUT-01, MUT-02, MUT-03?

**Assumption made:** Tests skip direct vectors for observability and mutation because:
1. Spec does not prescribe semgrep rule names for these (unlike SRP rules AR-01 through AR-06)
2. Spec treats them as WARN/non-blocking (lower priority than SRP/circular/async)
3. Implementation can add rules later; tests will verify violations are detected if rules emit them

**Confidence:** Medium — test suite is still complete for high-priority rules (SRP, circular, async). Observability/mutation tests can be added after rule definitions.

---

### 5. Exit Code Testing Deferred to Integration Level

**Would have asked:** How should we test exit codes (shadow mode exits 0, blocking mode exits 1)?

**Assumption made:** Exit code testing requires subprocess.run() CLI invocation, which is integration-level. Placed in `test_nfr_exit_codes` as placeholder. Actual exit code validation happens in gate_runner tests or CLI integration tests.

**Confidence:** High — gate_runner is responsible for setting exit codes; gate module just returns status dict. CLI-level tests verify propagation.

## Specification Gaps and Assumptions

### 1. Severity Mapping Thresholds (Spec Section 02)

Spec defines risk score computation:
```
risk_score = weighted_average(violations)
  CRITICAL=100, ERROR=80, WARN=50, INFO=10
```

**Assumption:** Implementation uses these weights. Tests verify weights are applied correctly (not hard-coded differently).

### 2. Architecture Score Computation (Spec Section 02)

Spec defines:
```
architecture_score = 100 - (AR_violations * 10), clamped [0, 100]
```

**Assumption:** Only violations with `rule_id` starting with "AR-" (AR-01 through AR-06) count toward this score. Tests verify score is in [0, 100] range.

### 3. Tool Invocation Order (Spec Section 03)

Spec specifies order: import-linter → eslint → semgrep → jscpd → radon.

**Assumption:** Implementation follows this order. Tests verify all tools are invoked, but do not enforce order (order is not observable from gate's output).

### 4. Deduplication Fingerprint (Spec Section 03)

Spec requires deduplication by `(file, line, rule_id)`.

**Assumption:** Implementation deduplicates using this fingerprint. Test `test_violation_deduplication` verifies same-fingerprint violations are merged.
