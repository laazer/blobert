### [M19-ARGLB] SPECIFICATION — octet-stream fallback for unknown extensions

**Would have asked:** The scaffold `assets.py` serves files with unknown extensions (not `.glb` or `.json`) using `application/octet-stream` as a media type fallback. The list endpoint, however, filters to only `.glb` and `.json` files (via `_MIME` key check). Should an unknown extension file reachable inside an export dir (placed manually) be served with `application/octet-stream` or rejected with 415?

**Assumption made:** The serve endpoint allows files with extensions not in `_MIME` and falls back to `application/octet-stream`. The list endpoint continues to filter to `.glb` and `.json` only. This asymmetry is intentional: serve is a general pass-through within the jail, but list only surfaces known types. The spec will document this explicitly.

**Confidence:** Medium

### [M19-ARGLB] SPECIFICATION — `refreshAssetsAndAutoSelect` hardcodes `activeAnimation = "Idle"`

**Would have asked:** In `useAppStore.ts`, `refreshAssetsAndAutoSelect` sets `s.activeAnimation = "Idle"` unconditionally when a matching asset is found. This means the active animation is always reset to "Idle" on each new generation, ignoring what the model actually contains. Should this be preserved or should the store reset `activeAnimation` to `null` and let `GlbViewer.Model` set the first real clip after load?

**Assumption made:** The spec should require that `refreshAssetsAndAutoSelect` sets `activeAnimation` to `null` on new model load, and that `GlbViewer.Model` auto-selects the first available clip from `names` via `setActiveAnimation` only if `activeAnimation` is null after the model loads. "Idle" is not guaranteed to exist in all GLBs. This matches the `availableClips` fix direction and is more correct. The spec will mandate this behavior.

**Confidence:** High

### [M19-ARGLB] SPECIFICATION — `AnimationControls` prop vs. store source of truth

**Would have asked:** `AnimationControls` accepts an optional `availableClips?: string[]` prop and falls back to the hardcoded 13-clip list. After the store gains `availableClips: string[]`, should the prop be removed to enforce store-only sourcing, or kept for backward compatibility?

**Assumption made:** The prop should be kept as an optional override to avoid breaking callers, but the spec will mandate that the component uses the store's `availableClips` slice as primary source when it is non-empty, and falls back to the prop (then to the hardcoded list) if both are empty. This is a superset of current behavior, not a regression.

**Confidence:** Medium

### [M19-ARGLB] SPECIFICATION — frontend ACs cannot be auto-tested in this pipeline

**Would have asked:** ACs 4–7 (auto-load after generation, OrbitControls, animation playback, ErrorBoundary display in canvas) require a browser rendering environment. Should these be marked as "manual verification" or deferred to a later Playwright sprint?
