## Checkpoint

### [M11-08] SPECIFICATION — EnemyBase.State integer literals in _apply_modifiers
**Would have asked:** Should `_apply_modifiers` reference `EnemyBase.State.WEAKENED` enum or use integer literal `1`?
**Assumption made:** Use integer literals (0=NORMAL, 1=WEAKENED, 2=INFECTED) because `_apply_modifiers` receives `target: Node3D` and cannot reference `EnemyBase.State` without a cast or import. The enum values are stable per `enemy_base.gd` line 23.
**Confidence:** High

### [M11-08] SPECIFICATION — Projectile path and infect_weakened
**Would have asked:** Should `PlayerProjectile3D` support `infect_weakened` modifier?
**Assumption made:** No. The `pre_damage_state` parameter defaults to `-1` in projectile calls (backward compat). Since projectile resources should not have `"infect_weakened": true`, this is a non-issue. If a projectile resource DID have it, it would fall back to current state (post-damage), which could produce same-hit weaken+infect — but this combination is not intended and not registered.
**Confidence:** High

### [M11-08] SPECIFICATION — Claw attack_id value
**Would have asked:** What should `attack_id` be for claw? Is there a numbering scheme?
**Assumption made:** Use `attack_id = 1` as the first registered base attack. Future mutations increment (acid=2, adhesion=3, carapace=4). The `attack_id` is an internal numeric identifier on AttackResource, not used for lookup (lookup uses mutation_id string).
**Confidence:** Medium

### [M11-08] SPECIFICATION — Knockback magnitude value
**Would have asked:** What knockback magnitude for claw? Ticket does not specify.
**Assumption made:** Use `knockback_magnitude = 2.0` (light push). Claw is a fast attack with short range — moderate knockback gives positional feedback without pushing enemies too far for the follow-up infection hit.
**Confidence:** Medium
