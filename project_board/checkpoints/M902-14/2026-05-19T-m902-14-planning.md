## Design Decisions (Checkpoint Log)

### [M902-14] PLANNING — Design Scope & Constraints

**Decision 1: Agent receives BUNDLE INPUT ONLY (no repo context)**

**Would have asked:** Should agent have access to git history, file system, or full repo to validate its evaluations?

**Assumption made:** Agent receives ONLY JSON bundle from M902-13. Agent has NO access to git, file system, full repo, or project context. All evaluation must use information present in bundle (code hunks, import graph, ownership, violations summary, etc.).

**Rationale:** Limiting agent input to focused bundle (1) reduces hallucination risk (no "unreliable" external context), (2) makes agent portable (can evaluate bundles offline), (3) enforces isolation per code_governance.md Stage 6 ("agents evaluate only extracted bundles"), (4) simplifies testing (mock bundles instead of git repos).

**Confidence:** HIGH. Spec freezes this constraint in Requirement 02 (input contract). Tests validate bundle-only input (no file/git access mocking).

---

**Decision 2: Agent decision logic DETERMINISTIC (rule-based, not LLM-based)**

**Would have asked:** Should agent use LLM-based reasoning (GPT, Claude, etc.) for semantic evaluation? These could provide richer reasoning but lose determinism.

**Assumption made:** Agent logic is RULE-BASED (not LLM-based). Same bundle always produces identical decision JSON (byte-for-byte after JSON normalization). No randomness, no temperature/sampling, no timestamps in decision logic.

**Rationale:** (1) Determinism required by test suite (adversarial tests validate idempotence), (2) Rule-based logic is transparent + auditable (decision tree clear), (3) Deterministic output supports reproducible CI/CD, (4) Avoids hallucination risk from LLM sampling.

**Tradeoff:** Rule-based logic is less "intelligent" than LLM-based reasoning. Evaluations may miss subtle architectural issues. Mitigated by: (a) evaluation rules grounded in code_governance.md (not invented), (b) human review remains available (Stage 7 override system), (c) future ML refinement deferred to M905+.

**Confidence:** HIGH. Spec Requirement 03 documents rule-based decision logic. Tests validate determinism (run twice, identical output).

---

**Decision 3: Decision priority CASCADING (reject > warn > approve)**

**Would have asked:** How should agent decide when multiple signals conflict (e.g., SRP violation + clear abstraction)? Should all signals be weighted equally?

**Assumption made:** Decision priority CASCADES per code_governance.md Stage 6 exit rules:
1. REJECT if ANY signal is CRITICAL (async safety violation is hardcoded REJECT)
2. WARN if ANY signal is AMBIGUOUS (observability gap, suppression without justification)
3. APPROVE if ALL signals are CLEAN (no violations, proper design)

**Rationale:** (1) Cascading priority aligns with code_governance.md Stage 6 exit rules ("reject async > reject SRP > warn observability"), (2) Prevents approval of fundamentally unsafe code (async violations are non-negotiable), (3) Encourages incremental fixes (warn for ambiguous, don't reject), (4) Simplifies implementation (independent signal evaluation, then merge decisions).

**Edge case:** Multiple violations at same level (e.g., SRP + abstraction both violations) → merge with OR logic (any violation at that level triggers decision).

**Confidence:** MEDIUM-HIGH. code_governance.md Stage 6 defines exit rules; Task 3 (Test Breaker) clarifies rule conflict order in checkpoint decisions.

---

**Decision 4: Confidence scoring HEURISTIC (not ML confidence)**

**Would have asked:** How should confidence score (0.0–1.0) be calculated? Should it reflect agent's certainty about the decision?

**Assumption made:** Confidence score reflects decision STRENGTH, not agent certainty:
- APPROVE with high confidence (0.8): clean bundle, no violations, clear design
- APPROVE with medium confidence (0.7): clean bundle, but minor inconsistencies noted
- WARN with confidence (0.5): ambiguous signals, needs human review
- REJECT with confidence (0.6): critical violation detected, clear fix required
- (Example values; exact thresholds frozen in Task 1 Spec)

**Rationale:** (1) Score reflects signal strength (number of violations, severity, clarity), not model uncertainty, (2) Usable for prioritization (high confidence → fast-track, low confidence → escalate), (3) Transparent to M903 orchestrator (score is artifact of rule evaluation, not "magical" ML confidence).

**Tradeoff:** Score is heuristic (hardcoded thresholds), not data-driven. Mitigated by: (a) thresholds grounded in code_governance.md rules, (b) can be tuned per project needs, (c) future ML refinement in M905+.

**Confidence:** MEDIUM. Confidence scoring heuristics frozen in Task 1 Spec (Requirement 03). Tests validate score bounds [0.0, 1.0].

---

**Decision 5: 8 evaluation SIGNALS organized as INDEPENDENT sub-evaluations**

**Would have asked:** How should agent evaluate 8 different signals? Should they be evaluated sequentially, in parallel, or as a single holistic evaluation?

**Assumption made:** 8 signals evaluated INDEPENDENTLY (separate function per signal or clearly demarcated code section):
1. SRP (Single Responsibility Principle)
2. Abstraction (abstraction justification, not over-engineering)
3. Hierarchy (inheritance depth, layering correctness)
4. Ownership (clear ownership assignments, no conflicts)
5. Observability (structured logging, audit events, correlation IDs)
6. Async (blocking in async context, cancellation, sync/async boundaries)
7. Exception (silent failures, proper re-raise, context preservation)
8. Suppression (justified suppressions, expiration, ticket references)

**Rationale:** (1) Independent evaluation simplifies logic (no cross-signal weighting), (2) Each signal can be tested in isolation (50+ behavioral tests = 8×5+ = 40+ signal tests), (3) Clear SRP at module level (each signal → one function), (4) Easier to extend (add 9th signal later without rewriting others).

**Merging:** After independent evaluation, signals are merged per decision priority (reject if critical, warn if ambiguous, approve if clean).

**Confidence:** HIGH. Spec Requirement 02 defines 8 signals with independent evaluation rules. Task 2 (Test Designer) organizes tests by signal type.

---

**Decision 6: Bundle-only EVALUATION (no access to prior gate outputs directly)**

**Would have asked:** Should agent re-evaluate signals from scratch, or should it trust prior gate outputs (M902-11 architecture, M902-12 risk_scoring)?

**Assumption made:** Agent evaluates bundles INDEPENDENTLY (re-evaluates signals using information in bundle). Bundle contains violations summary from prior gates, but agent does NOT invoke prior gates directly. Agent logic is stateless (no prior gate invocation, no caching).

**Rationale:** (1) Agent operates offline (bundle is JSON file, no runtime context), (2) Simplifies testing (mock bundle only, not entire gate pipeline), (3) Allows async evaluation (bundle can be evaluated hours later), (4) Supports offline code review (bundle can be shared/analyzed without repo).

**Tradeoff:** Agent may re-evaluate signals differently than prior gates (inconsistency risk). Mitigated by: (a) bundle includes prior gate violations (context for reconciliation), (b) human reviewer (Stage 7 override system) can override agent if inconsistency detected, (c) future integration refinement in M903+.

**Confidence:** HIGH. Spec Requirement 02 defines bundle-only input contract. Tests validate no prior gate invocation.
