### [M25-04] PLANNING — control definition placement: dedicated function vs. ClassVar promotion
**Would have asked:** Should `RIG_HEAD_ROT_X/Y/Z` and `RIG_BODY_ROT_X/Y/Z` be added as ClassVar floats on each enemy class (so they appear automatically via `_mesh_numeric_defaults()` introspection), or defined as a dedicated function like `_rig_rotation_control_defs()` added to `animated_build_controls_for_api()` and `coerce_validate_enemy_build_options()`?
**Assumption made:** Add a dedicated `_rig_rotation_control_defs()` function in `animated_build_options.py`. Reason: ClassVar introspection via `_mesh_float_control_defs()` generates bounds dynamically from the default value (`lo = max(0.01, min(vf * 0.05, vf * 0.5))`, `hi = max(5.0, vf * 5.0)`). For default=0.0 the generated range would be `[0.0, 2.0]` — wrong bounds for rotation. The `step` must be 1 degree (not 0.05). Explicit control defs give exact `min=-180, max=180, step=1, default=0` on all axes. The existing `_tail_control_defs()` / `_mouth_control_defs()` pattern is the correct precedent.
**Confidence:** High

### [M25-04] PLANNING — Blender rotation application: object.rotation_euler vs. mathutils
**Would have asked:** Should rotation be applied via `bpy.ops.transform.rotate`, `obj.rotation_euler`, or a helper in `blender_utils`? Which Euler order?
**Assumption made:** Apply rotation directly to the Blender mesh object using `obj.rotation_euler = mathutils.Euler((rx, ry, rz), 'XYZ')` where rx/ry/rz are in radians converted from the degree build option. This matches Blender's default Euler order and avoids bpy.ops overhead. The existing codebase does not have a generic mesh-rotation helper; no new helper is needed for this ticket — inline in `build_mesh_parts()` per slug.
**Confidence:** Medium (Blender API assumption; the Spec Agent should verify the exact rotation_euler API usage against the existing blender_utils/stubs pattern)

### [M25-04] PLANNING — `options_for_enemy` wiring: allowed_non_mesh vs. static_defs.extend
**Would have asked:** Should `RIG_HEAD_ROT_X` etc. be routed through the `allowed_non_mesh` set in `options_for_enemy()` (like tail/mouth controls), or wired through the `static_defs.extend()` path in `coerce_validate_enemy_build_options()`?
**Assumption made:** Wire through both:
1. `_rig_rotation_control_defs()` included in `static_defs.extend()` inside `coerce_validate_enemy_build_options()` for type coercion/clamping.
2. Keys added to `allowed_non_mesh` in `options_for_enemy()` so they pass through the merge step.
3. Keys added to `_defaults_for_slug()` so they serialize with default 0 when no value is supplied.
This exactly mirrors the pattern used for `_tail_control_defs()` and `_mouth_control_defs()`.
**Confidence:** High
