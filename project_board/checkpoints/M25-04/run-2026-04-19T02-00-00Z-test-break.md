## [M25-04] TEST_BREAK — player_slime in _defaults_for_slug() — conservative enforcement

**Would have asked:** PRC-3 Risk section explicitly says that `_defaults_for_slug()` may be
unconditional and that `player_slime` would therefore receive rotation keys in its defaults dict,
calling this "harmless for the defaults function". Should the test assert that player_slime does NOT
have rotation keys in its defaults, or should it allow the permissive behavior described in the risk section?

**Assumption made:** Enforce the stricter variant — player_slime must NOT have rotation keys in
`_defaults_for_slug()`. The spec's intent throughout is that rotation controls are exclusive to the 6
animated enemy slugs. Even if the defaults function is technically unconditional, exposing rotation keys
in player_slime defaults is a latent hazard: those keys could propagate into coerce_validate or API
responses via code paths not yet audited. The conservative assumption blocks this hazard. Marked with
`# CHECKPOINT` in the test file. If the implementer finds the conditional approach impractical,
this test should be discussed with the human before relaxing it.

**Confidence:** Medium

## [M25-04] TEST_BREAK — None input coercion — not in spec

**Would have asked:** PRC-5 covers NaN, inf, and non-parseable string. It does not mention `None`.
The existing validate loop uses `float(out[key])` which raises `TypeError` for `None`. Should `None`
be treated like a non-parseable string (default 0.0) or should it raise cleanly?

**Assumption made:** Write a test asserting `None` reverts to default 0.0. This is the safest
user-facing behavior. If the implementation raises a `TypeError` instead, the test surfaces the gap
so the implementation agent can explicitly handle `None` input. The test is marked `# CHECKPOINT`.
If the team decides that `None` should raise, the test expectation can be changed to `pytest.raises(TypeError)`.

**Confidence:** Medium

## [M25-04] TEST_BREAK — Ordering by index: contiguous rotation defs

**Would have asked:** AC-2.3/AC-2.4 assert ordering (after static_float, before mesh floats)
but do not explicitly require the 6 rotation defs to be contiguous. A broken implementation
could interleave them with other controls and still satisfy "appears after X and before Y"
individually. Should we test contiguity?

**Assumption made:** Yes — test that the 6 rotation keys appear at consecutive indices. An
implementation that inserts them non-contiguously would produce confusing UX in the web editor
and violates the implicit grouping intent of the spec.

**Confidence:** High
