## Analysis

### What Already Exists (verified via code inspection)

1. **AttackExecutor._handle_projectile_spit()** (`scripts/attacks/attack_executor.gd`): Creates `PlayerProjectile3D`, sets damage/speed/lifetime/knockback/modifiers/direction/color from AttackResource, adds to scene tree. Fully functional.
2. **PlayerProjectile3D._on_body_entered()** (`scripts/attacks/player_projectile_3d.gd`): On collision with an entity having `take_damage()`, calls `take_damage(damage, kb)` then `_apply_modifiers(body)`. Consumed flag prevents double-hit.
3. **PlayerProjectile3D._apply_modifiers()**: Handles `slow` modifier — calls `target.apply_slowness(slow_val, slow_duration)`. The seam for root is here.
4. **EnemyBase.apply_slowness(multiplier, duration)** (`scripts/enemies/enemy_base.gd`): Delegates to `_effect_tracker.set_slowness(multiplier, duration)`. Already supports multiplier=0.0 which would be a full root.
5. **EnemyEffectTracker.set_slowness()** (`scripts/enemies/enemy_effect_tracker.gd`): Clamps multiplier with `maxf(0.0, multiplier)`, tracks remaining duration. `get_speed_multiplier()` returns the active multiplier or 1.0 when expired.
6. **AttackDatabaseNode._register_defaults()** (`scripts/attacks/attack_database.gd`): Has claw (MELEE_SWIPE), acid (PROJECTILE_SPIT), carapace (SLAM_AOE) patterns. Adhesion follows the acid template.
7. **AttackResource** (`scripts/attacks/attack_resource.gd`): 15 exports including `projectile_speed`, `projectile_lifetime`, `color`, `modifiers`.
8. **Existing adhesion enemy test** (`tests/scripts/combat/test_adhesion_enemy_attack.gd`): M8 tests for *enemy→player* root timer. This is unrelated to M11-11 (player→enemy adhesion projectile).

### What's New (gaps to fill)

1. **Register adhesion AttackResource** in `_register_defaults()`: effect_type "PROJECTILE_SPIT", cooldown 2.5s, projectile range 10 units (→ `projectile_lifetime = range / speed`), adhesion-specific color.
2. **Full root effect via modifiers**: Set `slow` modifier to `0.0` (complete movement stop) with `slow_duration` of `1.0`. BUT: current `_apply_modifiers` checks `if slow_val and slow_val > 0.0` which will be falsy for 0.0. This is the **critical code change**: the slow check must be changed to handle `0.0` as a valid root.
3. **Root + infection interaction**: A rooted enemy (speed_multiplier == 0.0) should be easier to infect. This requires spec clarification on exact mechanic — likely a new modifier key or integration with existing infection flow.
4. **Wall collision despawn**: `PlayerProjectile3D._on_body_entered()` only fires for bodies with `take_damage()`. Wall bodies (StaticBody3D/WorldEnvironment) don't have `take_damage()` and are silently ignored. Need to add wall collision handling.
5. **Visual distinction**: Distinct color on projectile (e.g., `Color.DARK_GOLDENROD` or similar earthy/sticky color).

### Critical Bug/Change Identified

**`PlayerProjectile3D._apply_modifiers()` line 76-79:** The slow check uses `if slow_val and slow_val > 0.0`, which is falsy for `slow_val = 0.0`. A root (speed multiplier = 0.0) will SILENTLY NOT APPLY. This must be fixed to `if slow_val != null and typeof(slow_val) == TYPE_FLOAT` or similar truthy-safe check. The same issue exists in `AttackExecutor._apply_modifiers()` line 124-127.

## Checkpoint

### [M11-11] PLANNING — Root effect via slow modifier with value 0.0
**Would have asked:** Should root be implemented as a new `apply_root()` method on EnemyBase, or reuse `apply_slowness(0.0, 1.0)`?
**Assumption made:** Reuse `apply_slowness(0.0, 1.0)` via the existing `slow` modifier in modifiers dict. This avoids new EnemyBase API surface and matches the established modifier pattern. The blocker is that both `PlayerProjectile3D._apply_modifiers()` and `AttackExecutor._apply_modifiers()` treat `slow_val = 0.0` as falsy, which must be fixed. The fix is to change the truthy check to an explicit type/null check.
**Confidence:** High

### [M11-11] PLANNING — Root + infection interaction
**Would have asked:** What does "rooted enemy is easier to infect" mean mechanically? Options: (a) rooted+NORMAL→auto-infect, (b) rooted+WEAKENED→auto-infect (skip requirement of claw hit), (c) reduced HP threshold for WEAKENED during root.
**Assumption made:** The most conservative interpretation consistent with existing infection flow: a rooted enemy in NORMAL state that is *also* WEAKENED (HP ≤ 50%) can be infected by the adhesion projectile itself (add `infect_weakened` to adhesion modifiers). This extends the existing `infect_weakened` modifier without new mechanics. Spec Agent will finalize.
**Confidence:** Medium

### [M11-11] PLANNING — Wall collision mechanism
**Would have asked:** How should projectile detect walls? Via collision layer/mask on the Area3D, or via checking body type in `_on_body_entered()`?
**Assumption made:** Extend `_on_body_entered()` to handle bodies WITHOUT `take_damage()` — if the body is a StaticBody3D (wall/floor) or similar non-enemy, consume and queue_free. This avoids needing adhesion-specific collision masks and works for all PROJECTILE_SPIT projectiles. The existing acid projectile also benefits from this fix.
**Confidence:** High

### [M11-11] PLANNING — Projectile lifetime from range
**Would have asked:** Ticket says "projectile range 10 units" but AttackResource has `projectile_lifetime` (seconds), not range. How to derive?
**Assumption made:** `projectile_lifetime = range / projectile_speed`. With range=10 and speed=8 (matching acid's speed), lifetime ≈ 1.25s. Spec Agent determines exact speed; registration uses computed lifetime.
**Confidence:** High
