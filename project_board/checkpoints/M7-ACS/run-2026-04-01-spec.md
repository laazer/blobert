### [M7-ACS] Spec — velocity source: get_parent() vs owner

**Would have asked:** The planning checkpoint says "read `owner.velocity`" but
the controller is a child Node of the enemy root (a CharacterBody3D with
EnemyBase script). In Godot 4, `owner` is the scene root (set during
PackedScene packing), not necessarily the direct parent. Should the controller
read `get_parent().velocity` (always the immediate parent node) or
`owner.velocity` (scene root)? These could differ if the controller is nested
deeper.

**Assumption made:** Use `get_parent().velocity` because the planning decision
states the controller attaches "as a child node of the enemy root" — one level
deep only. `get_parent()` is deterministic in that topology. `owner` is
unreliable in headless tests where no PackedScene packing occurs. Spec uses
`get_parent()` everywhere velocity is referenced.

**Confidence:** High

### [M7-ACS] Spec — _process vs _physics_process for state polling

**Would have asked:** Should the controller poll EnemyStateMachine state in
`_process` (render frame) or `_physics_process` (fixed physics step)? The
parent CharacterBody3D updates `velocity` during `_physics_process`. Reading
velocity in `_process` could sample a stale value on the same frame.

**Assumption made:** Use `_physics_process` so velocity and state are read
after the parent's physics step completes. This eliminates one-frame lag on
state transitions. The spec mandates `_physics_process` as the polling hook.

**Confidence:** High

### [M7-ACS] Spec — Hit animation re-entrancy

**Would have asked:** If `trigger_hit_animation()` is called a second time
while the Hit clip is still playing, should the controller: (a) ignore the
call, (b) restart the Hit clip from frame 0, or (c) extend the hit?

**Assumption made:** Restart from frame 0 (option b). This is the most
responsive feel and matches common game animation conventions. The spec
requires that calling `trigger_hit_animation()` while already playing Hit
restarts the clip. The prior-clip bookkeeping is unaffected (prior clip
remains whatever was playing before the first hit call).

**Confidence:** Medium

### [M7-ACS] Spec — Death state re-entrancy

**Would have asked:** If `apply_death_event()` is called while Death is already
playing (EnemyStateMachine is already `dead`), should the controller re-start
the Death clip or no-op?

**Assumption made:** No-op. Once the state machine is `dead` and the Death clip
has been triggered, subsequent `_physics_process` ticks must detect that Death
is already playing (via AnimationPlayer stub `current_animation == "Death"`)
and skip re-triggering. This prevents animation reset mid-death.

**Confidence:** High

### [M7-ACS] Spec — what constitutes AnimationPlayer "stub" interface

**Would have asked:** The planning decision says tests use a stub AnimationPlayer.
What exact methods/properties does the stub need to expose so the controller
compiles cleanly against it (since GDScript uses duck typing for @export vars)?

**Assumption made:** The stub must implement: `play(anim_name: String,
blend_time: float) -> void`, `stop() -> void`, `is_playing() -> bool`,
`current_animation: String` (readable property). The controller must only call
these four surface points so the stub remains minimal. The spec enumerates
exactly these four.

**Confidence:** High

### [M7-ACS] Spec — prior-clip resume after Hit when state is dead

**Would have asked:** If the enemy dies while the Hit animation is playing, and
the Hit animation finishes, should the controller resume `prior_clip` (which
would play a looping state clip on a dead enemy) or check state again and
switch to Death?

**Assumption made:** On Hit completion, the controller must re-evaluate the
current state rather than blindly restoring `prior_clip`. The resume logic
calls the same state-to-clip resolution function. This means a dead enemy
whose Hit finishes will transition to Death, not loop a walk/idle clip.

**Confidence:** High

### [M7-ACS] Spec — blend_time for weakened/infected speed changes

**Would have asked:** When transitioning from `active`/`idle` to `weakened` or
`infected`, the clip name stays `"Idle"` but the playback speed changes. Does
the blend_time apply to this transition (i.e., does the controller call `play`
again with blend_time when only the speed changes)?

**Assumption made:** Yes. Whenever the resolved (clip_name, speed) tuple
changes — even if only speed differs — the controller calls `play` again with
`blend_time`. This prevents a hard pop when the enemy snaps to slow-motion idle.

**Confidence:** Medium
