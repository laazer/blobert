### [M7-ACS] Impl — hit-active vs death priority ordering in _physics_process

**Would have asked:** ADV-EAC-12 requires that death does NOT trigger while _hit_active is true (hit clip still playing). But ACS-7 says death is the first thing checked in the resolution table. Should the _hit_active suppression gate apply before the death latch check, or after?

**Assumption made:** The _hit_active suppression must be checked BEFORE the death branch fires. When _hit_active is true and is_playing() is still true (hit clip in progress), the controller must do nothing — including skipping the death latch. Only when the hit clip ends (is_playing()==false OR current_animation!="Hit") does the controller re-evaluate state (which may then trigger death). This is confirmed by ADV-EAC-12 expecting _death_latched=false while hit is still playing with state="dead".

**Confidence:** High

### [M7-ACS] Impl — _prior_clip and _prior_speed fields retained

**Would have asked:** ACS-6 notes that _prior_clip and _prior_speed may be "cleanup candidates" since AC-6.6 says resume uses live state re-evaluation, not the prior fields. But EAC-17 reads controller._prior_clip and controller._prior_speed directly. Should these fields exist in the implementation?

**Assumption made:** Yes. The fields must exist as internal state variables because the tests assert their values directly (EAC-17 checks that _prior_clip is set on first hit call and unchanged on re-entrant call). Even though resume uses live state, the prior fields are set when trigger_hit_animation() is called for the first time (not re-entrant).

**Confidence:** High

### [M7-ACS] Impl — trigger_hit_animation() guards _ready_ok

**Would have asked:** ADV-EAC-11 requires trigger_hit_animation() to be a no-op when _ready_ok is false. The spec only mentions _ready_ok gating _physics_process. Should the same guard apply to the public trigger_hit_animation() method?

**Assumption made:** Yes. With animation_player null (which causes _ready_ok=false), any call to animation_player.play() inside trigger_hit_animation() would crash. The method must guard _ready_ok (or equivalently check _death_latched first, then _ready_ok).

**Confidence:** High

### [M7-ACS] Impl — animation_player @export type must be Node, not AnimationPlayer

**Would have asked:** ACS-2 specifies `@export var animation_player: AnimationPlayer` but ACS-8 assumes duck-typing will allow stub injection at runtime. Godot 4.6.1 enforces @export type annotations at runtime — assigning an Object (stub) to an AnimationPlayer-typed export raises "Invalid assignment" SCRIPT ERROR and the property is left null. All 39 tests fail silently (return early on null). Should the export type be weakened to Node to allow stub injection, accepting that the editor inspector will show it as generic Node rather than AnimationPlayer?

**Assumption made:** Change `@export var animation_player: AnimationPlayer` to `@export var animation_player: Node`. This mirrors the `state_machine: Node` pattern used for the same reason. The controller only calls methods on the object (play, stop, is_playing, current_animation, speed_scale) — it never requires AnimationPlayer-specific type APIs. This is the only structural change that makes all 39 tests pass without modifying the test files. The in-editor inspector benefit of typing it AnimationPlayer is outweighed by the test infrastructure requirement.

**Confidence:** High
