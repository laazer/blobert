## Checkpoint Entries

### [M11-10] PLANNING — Radial query center: player position vs offset

**Would have asked:** Should the SLAM_AOE radial query center on the player's exact position, or offset by facing direction like MELEE_SWIPE?

**Assumption made:** Center on player's exact position (`_get_owner_position()`). A ground slam is omnidirectional by nature — the impact radiates from where the player stands, not from a point in front. This differs from MELEE_SWIPE which offsets by `facing * range * 0.5`.

**Confidence:** High

### [M11-10] PLANNING — Wind-up implementation via startup_frames

**Would have asked:** Should the 0.2s wind-up use the existing `startup_frames` field (12 frames at 60fps) or a new `startup_delay_seconds` float?

**Assumption made:** Use `startup_frames = 12` (0.2s at 60fps). The `AttackResource` already has `startup_frames: int` and the executor's existing timer conversion (`startup_frames / FRAMES_PER_SECOND`) handles this. No new field needed. The claw spec (CPA-5) confirms `startup_frames = 0` means instant; carapace uses `startup_frames = 12` for 0.2s delay.

**Confidence:** High

### [M11-10] PLANNING — Airborne slam deferral mechanism

**Would have asked:** Should the airborne deferral be handled inside `_handle_slam_aoe()` (executor polls `is_on_floor()` via parent) or in `PlayerController3D` (queues attack, fires on next grounded frame)?

**Assumption made:** Handle inside `_handle_slam_aoe()` within AttackExecutor. After the wind-up delay, if the player is airborne, the handler awaits landing using a polling loop on `get_tree().create_timer()` checks of `is_on_floor()` via parent. This keeps the deferral logic self-contained in the executor without modifying `PlayerController3D._try_attack()` flow. The executor's `_is_active` flag naturally prevents overlapping attacks during the wait. The Spec Agent must freeze maximum wait time to prevent infinite hangs (e.g., 3.0s timeout → cancel if player hasn't landed).

**Confidence:** Medium — Controller-side approach is also viable but more invasive. Spec Agent should evaluate both and freeze.

### [M11-10] PLANNING — VFX signal for slam

**Would have asked:** Should carapace reuse `melee_vfx_requested` signal or add a new `slam_vfx_requested` signal?

**Assumption made:** Add a new `slam_vfx_requested` signal. The slam has different visual semantics (radial ground impact vs directional swipe). A distinct signal allows future VFX systems to render slam-specific effects (ground crack, dust ring) vs melee-specific effects (slash arc). This follows the pattern of `projectile_fired` being distinct from `melee_vfx_requested`.

**Confidence:** Medium — Spec Agent should confirm.
