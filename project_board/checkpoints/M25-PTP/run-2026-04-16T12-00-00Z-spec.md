### [M25-PTP] SPECIFICATION — GLSL shader for gradient/spots/stripes: vertex vs fragment
**Would have asked:** Should the procedural texture be computed in the vertex shader or fragment shader? Are UV coordinates assumed to be available on all GLB meshes, or should world-space / object-space position be used?
**Assumption made:** All three ShaderMaterials use fragment shaders with `vUv` (standard Three.js UV attribute) as the primary coordinate input. Because some procedural enemies may lack UV maps, the implementation falls back to `vPosition` (object-space position passed as a varying) when `uv` attribute is absent. The spec prescribes that each ShaderMaterial declares both `vUv` and `vPosition` varyings and the fragment shader uses `vUv.x`, `vUv.y` for gradient/stripe direction and a hash of `vPosition` for the spots pattern. The spec does not prescribe a specific GLSL snippet — it prescribes the uniform names and observable output (visually distinct patterns).
**Confidence:** Medium

### [M25-PTP] SPECIFICATION — empty hex string color → THREE.Color conversion
**Would have asked:** When `texture_grad_color_a` (or any color field) is the empty string `""`, what color should be applied?
**Assumption made:** Empty string `""` is interpreted as white (`#ffffff`). The ticket's Risk table states: "caller must convert to THREE.Color" with `new THREE.Color("#" + hex)` and "fallback to white for empty string." The spec prescribes: if the hex field is `""`, use `new THREE.Color(1, 1, 1)` (white). If the hex field is a non-empty string that does not parse as a valid hex color, also fall back to white.
**Confidence:** High
