## Discrepancy Resolutions

### [M11-04] SPECIFICATION ŌĆö `range` ŌåÆ `attack_range`
**Would have asked:** Should we use `range` (per ticket/design spec) or `attack_range` (per existing scripts)?
**Assumption made:** Use `attack_range` ŌĆö all four existing enemy scripts use this name, and `range` shadows GDScript built-in `range()`. Task directive confirms this choice.
**Confidence:** High

### [M11-04] SPECIFICATION ŌĆö `knockback_direction` placement
**Would have asked:** Should `knockback_direction` be a top-level export or inside modifiers?
**Assumption made:** Top-level export per ticket AC. It is a core combat parameter, not an optional modifier. Task directive confirms.
**Confidence:** High

### [M11-04] SPECIFICATION ŌĆö `description` and `projectile_lifetime` inclusion
**Would have asked:** The ticket omits these but the design spec includes them. Include or omit?
**Assumption made:** Include both. `description: String = ""` (useful for HUD), `projectile_lifetime: float = 2.0` (required for projectile despawn). Low-cost additions that avoid future breaking changes. Task directive confirms.
**Confidence:** High

### [M11-04] SPECIFICATION ŌĆö SLAM_AOE vs CHARGE for carapace attacks
**Would have asked:** The existing `carapace_husk_attack.gd` is a linear charge, but the task directive specifies SLAM_AOE as the effect_type. Are these the same?
**Assumption made:** Use SLAM_AOE per task directive. The existing carapace script is a charge, but the AttackResource data model only stores the effect_type string ŌĆö the executor (M11-05) defines the handler semantics. If both charge and slam behaviors are needed later, CHARGE can be added as a separate effect_type without changing AttackResource.
**Confidence:** Medium
