## Key Test Design Decisions

### Decision 1: Black-Box Behavioral Testing
- **Assumption Made:** Tests should validate observable behavior (file existence, schema compliance, syntax validity) rather than internal implementation details.
- **Rationale:** Spec focuses on deliverables (configs, docs, registry entries, script interface); tests should verify these are present and correct, not how they're implemented.
- **Confidence:** High. Aligns with spec acceptance criteria and test realism guardrail.

### Decision 2: No Mocking of External Tools (Except When Specified)
- **Assumption Made:** Tests should not mock ruff, mypy, eslint invocations unless spec explicitly requires it for determinism.
- **Rationale:** Spec expects the gate script to invoke real tools; tests validate that configuration enables tool execution.
- **Confidence:** High. Real tool invocation is observable behavior; mocking would hide integration issues.

### Decision 3: Graceful Fallback for Optional Dependencies
- **Assumption Made:** PyYAML (for YAML validation) and tomllib/tomli (for TOML parsing) may not be available; tests should skip validation rather than fail.
- **Rationale:** Test environment may vary; graceful degradation improves test robustness.
- **Confidence:** Medium. Uses pytest.skip() for optional validation steps.

### Decision 4: Loose Config Validation (Not Strict Rule Sets)
- **Assumption Made:** Tests validate configs exist and have required sections, not that rule sets are "complete" or "optimal."
- **Rationale:** Spec focuses on tool integration, not on rule coverage; M903 will tune rule sets.
- **Confidence:** High. Aligns with spec deferral of enforcement and grandfathering to M903.

### Decision 5: Spec References in Tests (Not Assertion of Prose)
- **Assumption Made:** Tests should NOT assert that markdown text matches specific wording; instead, tests validate structure (sections exist, fields are present).
- **Rationale:** Workflow enforcement module mandates tests verify executable behavior, not prose. Tests check for "static analysis section in README" not "README says exactly X."
- **Confidence:** High. Follows test realism guardrail strictly.
