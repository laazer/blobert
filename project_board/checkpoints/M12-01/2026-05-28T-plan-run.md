## Discovery Summary

### What already exists (implementation complete)

- `scripts/attacks/attack_database.gd`: `register_fused_attack()` and `get_fused_attack()` are fully implemented. Key sort uses `_make_fused_key()` which alphabetically sorts the two IDs — confirmed order-independence.
- `scripts/player/player_controller_3d.gd` lines 445–482: `_try_attack()` already has the full fused path: both-slot check → `get_fused_attack(a_id, b_id)` → sorted combo cooldown key → fallback to base slot_a. Cooldown is applied to a single sorted-combo key (shared key), not to each slot independently.
- `tests/scripts/attacks/test_attack_database_controller_integration.gd` lines 558–615: `test_adb07_fused_when_both_slots` and `test_adb07_fused_fallback_to_base` both exist and exercise the pipeline.
- `tests/scripts/attacks/test_attack_database.gd`: ADB-5 and ADB-6 cover `register_fused_attack` / `get_fused_attack` thoroughly including order-independence and self-fusion rejection.

### What is MISSING (the gaps)

1. **Combo matrix test coverage**: No test exercises all 6 unordered real-mutation combos (claw+acid, claw+carapace, claw+adhesion, acid+carapace, acid+adhesion, carapace+adhesion). The AC checkbox "Tests validate combo matrix coverage (6 unordered combos)" is unchecked. Grep for `combo_matrix`, `claw.*acid.*carapace`, and `6.*combo` returns no results in tests/.
2. **Cooldown design decision not documented**: The current `_try_attack()` uses a single sorted composite key for fused cooldown (e.g., `"acid_claw"`). The ticket notes a design choice: "shared cooldown (both mutations on same timer) or independent (each slot independent)". The current code chose composite-key shared cooldown. This decision is undocumented in a spec — the Spec Agent must freeze and document it before tests are written.
3. **Both-slot gating under fused cooldown**: The current code checks `_mutation_cooldowns.get(cooldown_key, 0.0) > 0.0` where `cooldown_key` is the composite key. But the fallback path uses `cooldown_key = a_id` (slot_a only, even when slot_b is filled). This asymmetry is untested: is it intentional that a fused combo fallback uses only slot_a's cooldown key, not slot_b's? The spec must address this edge case.
4. **No spec document for M12-01**: Unlike M11 tickets that have `project_board/specs/` documents, M12-01 has no spec file. The Spec Agent must produce one.
