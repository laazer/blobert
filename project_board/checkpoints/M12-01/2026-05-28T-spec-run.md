## Frozen Design Decisions

All four required decisions are frozen:

1. **FADI-DD-1 (Composite-Key Shared Cooldown):** Single shared composite timer, no individual-slot timers on fused fire. Confirmed intentional.
2. **FADI-DD-2 (Fallback Key Asymmetry):** Slot A is primary attacker on fallback. Slot B cooldown not read or set. Confirmed intentional, not a bug.
3. **FADI-DD-3 (Order-Independence):** `get_fused_attack(a,b) == get_fused_attack(b,a)` guaranteed by alphabetical sort in `_make_fused_key()`.
4. **FADI-DD-4 (6-Combo Matrix):** claw+acid (`acid_claw`), claw+carapace (`carapace_claw`), claw+adhesion (`adhesion_claw`), acid+carapace (`acid_carapace`), acid+adhesion (`acid_adhesion`), carapace+adhesion (`adhesion_carapace`).

## Checkpoint Assumptions

### [M12-01] SPECIFICATION — Fallback key asymmetry: intentional or bug

**Would have asked:** Is the fallback `cooldown_key = a_id` (slot A only, ignoring slot B) in the both-slots-filled-no-fused case intentional game design, or an unintentional omission?
**Assumption made:** Intentional. Slot A is the primary attacker; slot B contributes only when a fused combo exists. Freezing as correct design per FADI-DD-2. No implementation change required.
**Confidence:** High (the planning checkpoint logged the same interpretation; the code is clean and deliberate, not accidental)

### [M12-01] SPECIFICATION — Individual vs composite cooldown independence for EC-3

**Would have asked:** Should a player be able to fire a fused attack even if the individual-slot cooldown for one of the two mutations is active from a prior base-attack fire?
**Assumption made:** Yes — individual-slot cooldowns and the composite-key cooldown are completely independent timers. Firing a base attack for `"claw"` does not block a future fused `"acid_claw"` fire. This follows directly from FADI-DD-1 (composite key is a separate entry). Document as FADI-EC-3 at Medium risk.
**Confidence:** High (the code implements it this way; no evidence to the contrary)
