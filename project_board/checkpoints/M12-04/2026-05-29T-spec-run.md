## Checkpoint Protocol Entries

### [M12-04] SPECIFICATION — Knockback 80.0 Rationale

**Would have asked:** The M12-02 placeholder had knockback=3.0 consistent with other melee attacks (claw=2.0, carapace=5.0). The ticket AC specifies 80.0 which is dramatically higher. Is this a typo or intentional?

**Assumption made:** Normative from ticket AC and confirmed by planning checkpoint which flagged this as a known deliberate choice. 80.0 is the per-hit combo knockback in the engine's internal force units. Documented explicitly in spec AC-DD-1 to prevent future reduction.

**Confidence:** High — planning checkpoint explicitly stated "Document knockback 80.0 as a deliberate choice."

---

### [M12-04] SPECIFICATION — Counter Reset Behavior in stop_all_effects()

**Would have asked:** Should `_acid_stack_counter` reset to 0 when `stop_all_effects()` is called, allowing the counter to be reused?

**Assumption made:** Counter must NOT reset in `stop_all_effects()`. Resetting would allow a rapid re-stack immediately after a stop to reuse `"acid_stack_0"`, potentially colliding with a tick processing a cleared entry in the same frame. Monotonic counter eliminates this class of race. Documented in AC-3k.

**Confidence:** High — the correctness argument is clear. The counter consuming a few extra integers over the life of a combat encounter has no practical cost.

---

### [M12-04] SPECIFICATION — _apply_combo_modifiers vs Extending _apply_modifiers

**Would have asked:** Should a new `_apply_combo_modifiers()` method be created, or should `_apply_modifiers()` be extended with a parameter to switch between stacking and non-stacking acid?

**Assumption made:** New `_apply_combo_modifiers()` is the minimal, non-breaking approach. Extending `_apply_modifiers()` with a boolean parameter would require updating all existing call sites (3 in attack_executor.gd), risking accidental behavior change. Parallel method with single-line divergence is safer.

**Confidence:** High — minimal targeted change preferred per CLAUDE.md.

---

### [M12-04] SPECIFICATION — combo_frame_interval in modifiers dict vs new @export field

**Would have asked:** Should `combo_frame_interval` be a new `@export` field on AttackResource (like `combo_hits`) or stored in the `modifiers` dict?

**Assumption made:** Stored in `modifiers` dict as `"combo_frame_interval": ACID_CLAW_COMBO_FRAME_INTERVAL`. Rationale: `combo_frame_interval` is a combo-specific parameter that does not apply to MELEE_SWIPE or other effect types. Putting it in `modifiers` avoids polluting the base schema with a field that is meaningless for 4 out of 5 effect types. `combo_hits` is different — it IS read by the handler and has a meaningful default of 1 for all resources.

**Confidence:** Medium-High — the distinction is judgment. If Test Designer or Implementer finds strong reason to make it an @export field, that is a valid override.
