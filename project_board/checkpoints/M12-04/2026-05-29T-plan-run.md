## 1. Planning Decisions (Key Decisions)

### Design Question 1: Stats Conflict — M12-02 Registration vs Ticket AC

**Would have asked:** The existing `attack_database.gd` registers acid_claw (attack_id 101) as `MELEE_SWIPE`, damage=4.0, cooldown=1.5, range=1.5, knockback=3.0. The ticket AC specifies MELEE_SWIPE_COMBO, damage=1.8/hit, cooldown=2.0, range=1.2, knockback=80.0. Which is normative?

**Assumption made:** The ticket AC is the authoritative specification for M12-04. The M12-02 registration was a placeholder ("Toxic Slash" single-swipe) meant to register the combo ID; M12-04 defines the full behavioral contract. The Spec Agent must update the database registration to match the M12-04 AC stats (overwrite existing acid_claw constants and modifiers). The M12-02 "Toxic Slash" single-swipe is superseded by the M12-04 "Venomous Shred" combo definition.

**Confidence:** High — the ticket explicitly states effect_type MELEE_SWIPE_COMBO, which is a new type not in the executor, and combo_hits=3. The M12-02 registration (single MELEE_SWIPE) was clearly a stub.

---

### Design Question 2: MELEE_SWIPE_COMBO — New Effect Type or Reuse MELEE_SWIPE?

**Would have asked:** Should MELEE_SWIPE_COMBO be a new case in `execute_attack()`'s match statement, or should the single MELEE_SWIPE handler be extended to support a combo_hits modifier?

**Assumption made:** Implement MELEE_SWIPE_COMBO as a **new case in the match statement** (`"MELEE_SWIPE_COMBO": _handle_melee_swipe_combo(resource)`). This is cleaner, preserves the existing MELEE_SWIPE behavior (no regression), and matches the ticket's explicit effect_type string. The combo handler loops N times (combo_hits) with startup frame timing between swipes.

**Confidence:** High — the existing pattern (three discrete match cases) is well-established. Adding a fourth case is the minimal, non-invasive approach.

---

### Design Question 3: AcidVFXSystem Does Not Exist

**Would have asked:** The ticket says "apply unique acid DoT via AcidVFXSystem" and "use AcidVFXSystem.apply_acid() three times." No such class exists. What is the correct approach?

**Assumption made:** `AcidVFXSystem` is a **ticket-level fiction** — the ticket author intended to describe the acid DoT system conceptually. The actual implementation target is `enemy.apply_acid(duration, dps)` (which delegates to `EnemyEffectTracker.add_dot("acid", ...)`). No new `AcidVFXSystem` class will be created. The Spec Agent should document this interpretation and define the stacking mechanism using the existing enemy API.

**Confidence:** High — no AcidVFXSystem exists anywhere in the codebase (`scripts/`, `tests/`, `scenes/`). The existing acid system is `enemy_base.gd::apply_acid()`.

---

### Design Question 4: Stacking Acid DoT — EnemyEffectTracker Overwrites by Key

**Would have asked:** `EnemyEffectTracker.add_dot("acid", ...)` stores by key `"acid"` and overwrites on repeat calls. The ticket requires 3 independent stacks that decay independently. Does this need a new stacking mechanism, or can the test use a different API?

**Assumption made:** A new stacking mechanism is required. The Spec Agent must define whether to implement stacking by (a) using **indexed keys** (`"acid_0"`, `"acid_1"`, `"acid_2"`), (b) adding a dedicated `add_dot_stack(effect_name, duration, dps)` API to `EnemyEffectTracker` that auto-generates unique keys, or (c) verifying total acid damage (3×dps×duration) rather than stack count. The test AC `enemy.acid_stacks.size() == 3` implies an `acid_stacks` array property that does not currently exist.

**Assumption made (conservative):** Spec Agent should define a new `add_acid_stack(duration, dps)` method on `EnemyBase`/`EnemyEffectTracker` that stores stacks in a list (indexed by counter), and expose `get_acid_stack_count()` for test assertions. This is the minimal change that satisfies the AC. The Spec Agent must decide the exact API surface.

**Confidence:** Medium — the exact stacking API shape requires spec-level decision. The fact that a new mechanism is needed is HIGH confidence.

---

### Design Question 5: AttackResource Missing combo_hits Field

**Would have asked:** The ticket AC requires `combo_hits: 3` on the resource. `AttackResource` has no such field. Should it be added as a proper @export field or stored in the modifiers dictionary?

**Assumption made:** Add `combo_hits: int = 1` as a proper `@export` field on `AttackResource`. This is cleaner than stuffing it in `modifiers`, aligns with the existing pattern (attack_range, startup_frames are all explicit @export fields), and is required for the executor to read it type-safely. The Spec Agent must freeze this decision.

**Confidence:** Medium-High — explicit field is preferable. Using `modifiers` dict would work but is weaker. The Spec Agent may override if there's a strong reason to avoid modifying AttackResource (backward compat).

---

### Design Question 6: attack_id 101 vs 109 Discrepancy

**Would have asked:** The M12-02 ticket "Example" section shows `attack_id: 201` and the M12-04 ticket "Example Resource" (if any) showed 109. The actual database uses 101. Which is authoritative?

**Assumption made:** attack_id 101 as registered in `attack_database.gd` is authoritative. The M12-02 example numbers were non-normative (explicitly marked "Example" and noted as out-of-range by the previous Planner). The FADI spec defines the 6 fused combo IDs as 101–106. No ID conflict exists; 101 is the canonical acid_claw ID.

**Confidence:** High — confirmed from `attack_database.gd` line 33: `const ACID_CLAW_ATTACK_ID := 101` and the FADI spec's combo matrix.

---

### Design Question 7: Hitbox Frame Timing (frames 6, 12, 18)

**Would have asked:** The ticket specifies hitbox spawns at startup frames 6, 12, 18. These are inter-hit timings, not startup-before-first-hit. Does startup_frames on the resource apply to the first hit, and subsequent hits use a fixed inter-hit interval?

**Assumption made:** `startup_frames` (currently 0 in the M12-02 registration) applies to the pre-combo startup delay before the first swipe. The inter-hit delays (frames 6→12→18, i.e., 6 frames = 0.1s at 60fps between each swipe) are controlled by a new `combo_frame_interval: int` field or a constant in the handler. The Spec Agent should freeze the exact inter-hit timing mechanism (modifiers dict entry vs new field vs constant).

**Confidence:** Medium — the ticket is somewhat ambiguous about whether 6/12/18 are absolute frame counts from combo start or relative inter-hit intervals. Most natural reading: 3 hits, each 6 frames apart (0.1s at 60fps).
