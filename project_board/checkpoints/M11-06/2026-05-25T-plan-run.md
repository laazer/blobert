## Checkpoint Entries

### [M11-06] PLANNING — attack InputMap action registration

**Would have asked:** The `attack` action is defined in `PlayerInputActionPolicy` (IAM spec) but not yet registered in `project.godot` `[input]`. Should M11-06 register the InputMap action, or is that a separate ticket?

**Assumption made:** M11-06 must register the `attack` InputMap action in `project.godot` because the ticket AC requires `_try_attack()` to check `Input.is_action_just_pressed("attack")`, which requires the action to exist. IAM spec deferred this to "first code ticket that adds PlayerInputActionPolicy and/or new InputMap entries." M11-06 is that ticket.

**Confidence:** High

### [M11-06] PLANNING — AttackDatabase as autoload vs injected dependency

**Would have asked:** The ticket AC says "AttackDatabase autoload created." Should this be a true Godot autoload (registered in project.godot), or a manually instantiated node added as a child of PlayerController3D?

**Assumption made:** True autoload per ticket AC. Register in `project.godot` `[autoload]` section as `AttackDatabase="*res://scripts/attacks/attack_database.gd"`. This allows global access from any script.

**Confidence:** High

### [M11-06] PLANNING — .tres file loading vs code registration

**Would have asked:** Ticket says "AttackDatabase can load from code (in _register_base_attacks) or .tres files (decide in implementation)." Which approach?

**Assumption made:** Hybrid approach for spec to freeze: (1) provide a code-based `register_base_attack()` / `register_fused_attack()` API for tests and programmatic setup; (2) optionally scan `res://attacks/base/` and `res://attacks/fused/` directories on `_ready()` if they exist. This allows tests to inject resources without filesystem and supports .tres files in production. If no .tres files exist at launch, the database simply starts empty and attacks are registered in code.

**Confidence:** Medium

### [M11-06] PLANNING — GameState.get_active_mutation() mapping

**Would have asked:** Ticket pseudocode uses `GameState.get_active_mutation()` but no such global exists. The existing system uses `_mutation_slot` (from `InfectionInteractionHandler`). How should active mutation be determined?

**Assumption made:** Use the existing `_mutation_slot` system already in `PlayerController3D`. The `_is_mutation_active()` method already exists. For attack selection, the spec must define how `_mutation_slot` maps to a mutation_id for `get_base_attack(mutation_id)`. Spec Agent should freeze this mapping.

**Confidence:** Medium

### [M11-06] PLANNING — AttackExecutor node ownership

**Would have asked:** AttackExecutor is a Node subclass. Should it be a child of PlayerController3D, a sibling, or obtained from the autoload?

**Assumption made:** AttackExecutor should be instantiated as a child of PlayerController3D (like how _simulation and _player_state_machine are instantiated in `_ready()`). This gives it access to the player's position and facing direction via `get_parent()`, which its `_get_owner_position()` and `_get_facing_sign()` methods already expect.

**Confidence:** High
