## Checkpoint Entries

### [M11-06] SPECIFICATION — mutation_id type resolution

**Would have asked:** The ticket uses `mutation_id: int` in method signatures, but the existing mutation slot system (MutationSlot, EnemyInfection3D, InfectionAbsorbResolver) uses `String` throughout. Should the spec use int (per ticket) or String (per codebase)?

**Assumption made:** Use **String** for mutation_id to match the existing codebase. Evidence: `MutationSlot._active_mutation_id: String`, `EnemyInfection3D.mutation_drop: String = "infection_mutation_01"`, `InfectionAbsorbResolver.resolve_absorb(mutation_id: String)`. Using int would require a breaking change to the slot system. The ticket's int references are interpreted as illustrative pseudocode, not prescriptive.

**Confidence:** High

### [M11-06] SPECIFICATION — fused key format and order independence

**Would have asked:** Should `get_fused_attack("claw", "acid")` and `get_fused_attack("acid", "claw")` return the same result, or are slot positions significant?

**Assumption made:** Lookup is **order-independent**. Canonical key uses alphabetically sorted pair joined by underscore (e.g., `"acid_claw"`). The ticket says "key like slot_a_slot_b" which implies a fixed order, but for usability and correctness, order independence is preferred — the caller should not need to know which mutation was absorbed first.

**Confidence:** High

### [M11-06] SPECIFICATION — attack input binding J vs F

**Would have asked:** The ticket planning notes reference "F key per IAM-3.1" for the attack action, but F is already bound to `infect` in project.godot. The actual IAM spec table designates J. Which binding?

**Assumption made:** **J key** (physical keycode 74) per the frozen IAM spec table (IAM-3.1, line 219). The "F key" in the planning notes is an error; F (physical keycode 70) is already bound to `infect`.

**Confidence:** High

### [M11-06] SPECIFICATION — attack input suppression during movement root

**Would have asked:** Should attack input be suppressed when `_enemy_movement_root_remaining > 0.0` (enemy adhesion lunge root), similar to how movement and jump are suppressed?

**Assumption made:** Attack input is **NOT suppressed** during movement root. Movement root suppresses horizontal movement and jump (physical displacement) but not combat actions. The player should be able to attack while rooted — this is a common game pattern where roots prevent movement but not abilities.

**Confidence:** Medium

### [M11-06] SPECIFICATION — fused attack fallback behavior

**Would have asked:** When both mutation slots are filled but no fused attack is registered for that combo, should the player be unable to attack, or should it fall back to a base attack?

**Assumption made:** **Fall back to slot A's base attack**. This ensures the player always has an attack available when a mutation is active, even if the fused combo hasn't been registered yet. This is especially relevant since M11 only implements base attacks — fused attacks are M12.

**Confidence:** High

### [M11-06] SPECIFICATION — PlayerInputActionPolicy instantiation

**Would have asked:** PlayerController3D currently does not instantiate a PlayerInputActionPolicy. Should _try_attack() create one or use the permit matrix directly?

**Assumption made:** PlayerController3D should hold a `_input_policy: PlayerInputActionPolicy` instance (created in `_ready()`). This follows the existing pattern of creating helper objects in _ready() (like `_player_state_machine`). The policy provides the canonical state→action permission check and keeps the controller from duplicating the matrix logic.

**Confidence:** High
