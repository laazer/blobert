## Design Decisions Frozen

1. **Health Ratio Calculation** — `current_hp / max_hp` assuming both are > 0 by invariant (no defensive clamping; spec documents precondition).
2. **Auto-Hide Timeout** — Configurable float export (e.g. 3.0 seconds) on health bar script, not hardcoded.
3. **Camera-Facing** — Godot's native CanvasLayer with billboard mode or manual look_at() update per frame; spec will choose.
4. **Lifecycle Binding** — Health bar parent is enemy node; when enemy freed/despawned, health bar freed automatically (parent-child cleanup).
5. **Debug Flag Location** — `ProjectSettings.get_setting("blobert/debug/show_health_bars")` (preferred) or scene-level export (deferred to Spec Agent).
6. **UI Layer** — Separate 3D CanvasLayer from game 3D scene (no HUD overlay; world-space bars only).

## Ambiguities Resolved (Checkpoint Protocol)

### Q1: How is "current HP" provided to the health bar?
**Would have asked:** Where does the health component live? EnemyBase or separate script?
**Assumption made:** Health is managed by a health component (not yet implemented) or exposed via EnemyBase properties (current_hp, max_hp). Spec Agent will clarify in spec; implementation assumes EnemyBase or a wired signal from a health manager.
**Confidence:** MEDIUM (health system likely not implemented yet; spec must define contract).

### Q2: What triggers "recently damaged" visibility re-show?
**Would have asked:** Is this a damage signal or does the health bar watch current_hp every frame?
**Assumption made:** Health bar watches current_hp every frame; if current_hp < max_hp, bar is visible and timeout is reset. If current_hp == max_hp AND timeout expires, bar hides. No explicit damage event needed.
**Confidence:** HIGH (cleaner, frame-driven logic).

### Q3: What is the auto-hide timeout value?
**Would have asked:** Is 3 seconds hardcoded or configurable?
**Assumption made:** Configurable export float (default 3.0 seconds). Not hardcoded; allows design iteration.
**Confidence:** HIGH (matches project patterns).

### Q4: Can the health bar be toggled off entirely for performance?
**Would have asked:** Is a debug/project flag sufficient, or do we need runtime enable/disable per-enemy?
**Assumption made:** Single global `ProjectSettings` flag controls all bars. Per-enemy toggle deferred to future ticket (AC3 says "feature can be toggled," not "per-enemy").
**Confidence:** MEDIUM-HIGH (AC3 is phrased as singular feature, not granular control).

### Q5: How does the bar handle enemies with no health component?
**Would have asked:** Should the health bar fail silently, or should implementation error on missing health?
**Assumption made:** If health component is missing, the test suite will cover that as an adversarial case (missing current_hp/max_hp properties → bar should not render or should error in Godot console with clear message). Spec will define error handling contract.
**Confidence:** MEDIUM (adversarial suite will encode contract).
