### 2. Severity Thresholds & Decision Matrix (AC-4, AC-5)
**Decision:** Three-tier decision system with priority cascade.

| Finding Type | Severity | Decision | Exit Code |
|---|---|---|---|
| Any secret detected | CRITICAL | HARD FAIL | 1 (blocking) |
| Unsafe deserialization (B301-B303, pickle/yaml/marshal) | CRITICAL | HARD FAIL | 1 |
| Auth bypass patterns | CRITICAL | HARD FAIL | 1 |
| CVSS ≥7.0 CVE | HIGH/CRITICAL | HARD FAIL | 1 |
| CVSS 5.0–6.9 CVE | MEDIUM | WARN (soft fail) | 0 (shadow) |
| CVSS <5.0 CVE | LOW | WARN (soft fail) | 0 (shadow) |
| No critical findings | — | PASS | 0 |

**Rationale:** CVSS 7.0 is industry-standard critical/high boundary; hard-fail on secrets/auth/deserialization aligns with code_governance.md and OWASP principles; soft-fail on medium allows pipeline continuation in M902 shadow mode.

**Checkpoint Entry:**
```
### M902-16 PLANNING — Severity Thresholds
**Would have asked:** Should medium CVEs block in M903 enforcement? What about CVSS score variations across databases?
**Assumption made:** CVSS 7.0 threshold is fixed (corresponds to CVSS v3.1 scale); soft-fail on CVSS <7.0 is advisory; M903 enforces blocking logic. CVSS scores come from pip-audit/npm audit (both use NVD/Snyk databases).
**Confidence:** MEDIUM — Threshold is standard, but M903 may adjust. Spec will freeze exact values; implementation locked to spec.
```

### 3. Tool Ordering & Fail-Fast (AC-4)
**Decision:** Execute tools in order: gitleaks → bandit → semgrep → pip-audit → npm audit. Return HARD FAIL immediately on any secret detected (don't continue to other tools).

**Rationale:** Secrets are critical and must fail fast; other findings can be aggregated from remaining tools. Avoid wasting time scanning code for vulns if secrets are already leaked.

**Checkpoint Entry:**
```
### M902-16 PLANNING — Tool Execution Order
**Would have asked:** Should all tools always run in parallel or serial order?
**Assumption made:** Serial execution (gitleaks → bandit → semgrep → pip-audit → npm audit) with hard-fail-fast on secrets. Parallel execution would improve performance but complicates timeout management and result aggregation. Serial is safe default for M902 shadow mode.
**Confidence:** MEDIUM — Can optimize to parallel in M903 if performance is bottleneck; serial is deterministic and testable now.
```

### 4. Fixture & Test Strategy (AC-8, AC-9)
**Decision:** Create mock fixtures for all vulnerability types; never commit real secrets/exploits; test harness validates detection without introducing security risks.

- **Mock Secrets:** Fake AWS key patterns, GitHub tokens, RSA private keys, Stripe API keys (all non-functional)
- **Unsafe Code Patterns:** Python files with pickle.loads(), yaml.load(), eval() (valid syntax, flagged by bandit/semgrep)
- **Vulnerable Packages:** Test dependency lists referencing packages with known CVEs (isolated test environments, not main dependencies)

**Rationale:** Real vulnerabilities/secrets are dangerous and must never be in repo. Mock fixtures validate tool detection accuracy. Test harness uses temporary staging/venv isolation.

**Checkpoint Entry:**
```
### M902-16 PLANNING — Fixture Safety
**Would have asked:** Can we use real vulnerable packages for testing? What prevents accidental real secrets?
**Assumption made:** All fixtures are mock/non-functional. Test harness creates temporary isolated environments for dependency audit testing. Code review + gitleaks pre-commit hook prevents accidental real secret storage in repo.
**Confidence:** HIGH — Pattern established in M902-02 (static analysis fixtures); validated approach.
```

### 5. Determinism Requirement (AC-9)
**Decision:** Gate must produce identical findings for identical staged files every run. No randomness, no network drift, no tool version sensitivity (beyond pinned versions).

**Rationale:** Determinism is required for reproducible CI/security assessment; mandated by code_governance.md and M902-01 gate framework.

**Implementation Approach:**
- Tool versions pinned in pyproject.toml / package.json / tool configs
- No randomness in gate logic (rule-based, not LLM-based)
- No timestamps in decision logic (timestamp for metadata only)
- Offline mode for tools where available (no network fetch per run)
- Same violation order per tool run (deterministic sort if needed)

**Checkpoint Entry:**
```
### M902-16 PLANNING — Determinism Strategy
**Would have asked:** How strictly should determinism be enforced? Identical findings or semantically equivalent?
**Assumption made:** Identical JSON output required (same inputs → identical output bytes, not just semantic equivalence). Tests validate idempotence (run twice, get identical results).
**Confidence:** HIGH — Matches M902-01 gate framework contract and code_governance.md requirement.
```

### 6. Non-Blocking Shadow Mode in M902 (AC-5)
**Decision:** Gate runs in shadow mode in M902 (exits 0 regardless of findings). Hard-fail conditions return FAIL status (logged) but do not block CI. Enforcement (exit 1 on FAIL) deferred to M903.

**Rationale:** code_governance.md and M902 architecture call for advisory gates in M902, enforcement in M903. Shadow mode allows pipeline to continue while collecting security findings for review. Enforcement enables blocking in M903 after operator approval.

**Checkpoint Entry:**
```
### M902-16 PLANNING — Shadow Mode Scope
**Would have asked:** Should security gate have special enforcement in M902 (blocking on secrets even in shadow mode)?
**Assumption made:** Security gate follows standard M902 shadow mode pattern: advisory, non-blocking. Secrets are critical but must allow graceful failure path (audit + remediation) rather than hard-block. M903 can enable hard-blocking via enforcement toggle.
**Confidence:** MEDIUM — Aligns with M902 architecture, but security may warrant special handling. Spec will clarify; implementation will respect spec.
```
