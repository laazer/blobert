## Analysis

### What Already Exists (verified via code inspection)

1. **EnemyEffectTracker** (`scripts/enemies/enemy_effect_tracker.gd`): Full DoT tick system with 0.5s interval, non-stacking (dictionary key overwrite on `add_dot`).
2. **EnemyBase.apply_acid(duration, dps)**: Delegates to `_effect_tracker.add_dot("acid", duration, dps)`.
3. **AttackExecutor._apply_modifiers()**: Handles `acid_on_hit` modifier — calls `target.apply_acid(acid_duration, acid_dps)`.
4. **PlayerProjectile3D._apply_modifiers()**: Same `acid_on_hit` logic on projectile hit.
5. **AttackDatabaseNode._register_defaults()**: Already has claw registration pattern.
6. **PlayerProjectile3D**: Full lifecycle (movement, collision, consumed flag, despawn).

### What's New (gaps to fill)

1. **Register acid AttackResource** with PROJECTILE_SPIT effect_type, cooldown 2.0s, acid_on_hit modifier, 3.0s acid_duration, correct DPS.
2. **WEAKENED doubles DoT duration**: Currently neither `_apply_modifiers` nor `apply_acid` check target state. A pre-apply state check is needed (similar to `infect_weakened` pattern with `pre_damage_state`).
3. **Visual differentiation**: Acid projectile needs a distinct color (e.g., acid green vs adhesion's color). The `AttackResource.color` field exists but `PlayerProjectile3D` doesn't currently apply it visually.

### [M11-09] PLANNING — WEAKENED doubling location
**Would have asked:** Should the WEAKENED duration doubling check live in the projectile/executor `_apply_modifiers()` (checking target state before calling `apply_acid`) or inside `enemy_base.apply_acid()` itself?
**Assumption made:** Place it in `_apply_modifiers()` of both `PlayerProjectile3D` and `AttackExecutor`, matching the established pattern where modifier logic inspects target state (as claw does with `pre_damage_state` for `infect_weakened`). This keeps `apply_acid()` a simple pass-through and avoids coupling enemy logic to attack-specific mechanics.
**Confidence:** High
