## Decisions and Assumptions Logged

### [M8-EFHB] SPECIFICATION — Health Component Ownership
**Would have asked:** Should the health component be a child node of the enemy or embedded in the enemy script?  
**Assumption made:** Placed as a child node (`EnemyInfection3D/HealthComponent`) for separation of concerns and independent observability by the health bar and other systems.  
**Confidence:** High

---

### [M8-EFHB] SPECIFICATION — World-Space UI Rendering Technique
**Would have asked:** SubViewport, MeshInstance3D with shader, or custom billboard node?  
**Assumption made:** Spec defines behavior (world-space, camera-facing, updates HP ratio) without mandating the rendering approach. Test Designer and Implementation agent choose the technique that best fits Godot 4.x capabilities.  
**Confidence:** High

---

### [M8-EFHB] SPECIFICATION — Health Bar Visibility Timeout Duration
**Would have asked:** Exact timeout before hiding full-health bar?  
**Assumption made:** Timeout is an implementation detail; spec says "short timeout" and defines behavior. Implementation chooses a reasonable value (0.5s–1.0s). Test Designer parameterizes for robustness.  
**Confidence:** Medium (timeout value left as implementation choice per CLAUDE.md conventions)

---

### [M8-EFHB] SPECIFICATION — Default Max HP and Per-Enemy Variation
**Would have asked:** Do enemies have family-specific max HP or uniform max HP?  
**Assumption made:** Uniform max HP (100.0) for all enemies in M8. Per-family tuning is out of scope; can be added later via export variables.  
**Confidence:** Medium (no HP balance data in current enemy code)

---

### [M8-EFHB] SPECIFICATION — Heal Mechanics in Enemy Context
**Would have asked:** Do enemies ever heal in current gameplay? Is healing method required or aspirational?  
**Assumption made:** Healing is not a current M8 gameplay mechanic. Health component supports `take_heal()` for future extensibility and test completeness, but is not called by game logic in this milestone.  
**Confidence:** Medium (no existing heal signal in player or enemy code)

---

### [M8-EFHB] SPECIFICATION — Camera Frustum Culling
**Would have asked:** Should health bar render off-screen or be culled?  
**Assumption made:** Standard Godot camera culling applies. No special logic needed. Out-of-frustum bars are culled like enemy meshes.  
**Confidence:** High

---

### [M8-EFHB] SPECIFICATION — Multiple Health Bars Accumulation
**Would have asked:** How to prevent orphan health bars if enemies respawn or are cloned?  
**Assumption made:** Single bar created in enemy `_ready()`, freed on enemy death/despawn. Cleanup signal wiring prevents accumulation.  
**Confidence:** High (standard Godot lifecycle pattern)

---

### [M8-EFHB] SPECIFICATION — Debug Toggle Granularity
**Would have asked:** Global toggle or per-enemy?  
**Assumption made:** Global toggle via ProjectSettings (`debug/enemy_health_bar_enabled`). Per-enemy toggles are out of scope for M8.  
**Confidence:** High (ticket says "project/debug flag" → global)

---

### [M8-EFHB] SPECIFICATION — Health Bar Position Offset
**Would have asked:** Exact offset value; will it work for all enemy models?  
**Assumption made:** Default offset 1.2m above enemy root (reasonable for typical humanoid-scale enemies). Per-enemy or per-family offsets can be tuned later via export variables.  
**Confidence:** Medium (no asset metadata on exact enemy model heights)

---

### [M8-EFHB] SPECIFICATION — Frame Synchronization (Damage Signal Timing)
**Would have asked:** If damage occurs in `_physics_process` and bar updates in `_process`, can there be a one-frame delay?  
**Assumption made:** One-frame delay is acceptable; tests verify consistency within one frame. Godot's frame ordering ensures signal propagates before next `_process` callback.  
**Confidence:** High (standard Godot event flow)
