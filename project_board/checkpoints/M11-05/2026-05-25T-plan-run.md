### Key design decisions for Spec Agent

1. **AttackExecutor scope**: Node type (extends what?), scene-tree placement (child of PlayerController3D?), or standalone autoload?
2. **Enemy damage interface**: The ticket assumes `enemy.take_damage(damage, knockback_vector)`. This method doesn't exist on `EnemyBase`. Options:
   - Add `take_damage()` to `EnemyBase` as part of this ticket scope (tight coupling risk).
   - Define only the executor's *outbound call* contract and mock enemy API in tests — deferring the actual enemy method to a later ticket.
   - The spec should document which approach applies.
3. **Modifier application**: Ticket notes `apply_poison()`, `apply_acid()`, `apply_slowness()` — none exist. Executor should call them if present (duck-type / `has_method()` check) or use modifiers dict passthrough.
4. **Projectile scene**: Reuse `AcidProjectile3D` pattern but player-facing (damages enemies on hit). New class or generic `PlayerProjectile3D`?
5. **Startup frames timing**: Frame-based wait via `await get_tree().create_timer(frames / 60.0)` or `_physics_process` countdown? Must be testable.
6. **VFX / SFX**: Ticket mentions `spawn_vfx()` and `play_sfx()`. Are these real methods, or stubs that emit signals / call placeholder functions?
