### [M25-ESPS] PLANNING — scope of enemy types affected

**Would have asked:** Should eye shape/pupil controls be added only to enemies that already have eye geometry (spider, slug, imp, claw_crawler, carapace_husk, spitter) or only a subset?

**Assumption made:** All animated enemies that currently create explicit eye mesh parts (spider via `create_sphere` per-eye; slug via stalk + sphere eyes; claw_crawler via peripheral `create_sphere` calls) receive the new `eye_shape` / `pupil_*` controls. Enemies without explicit eye geometry (imp, spitter, carapace_husk — heads but no dedicated eye sub-mesh) receive controls but the Python builder falls back to the default sphere if the controls have no geometric impact (i.e., eye mesh objects do not exist). Conservative choice: gate controls per-enemy slug at the Python layer so only slugs with explicit eye mesh benefit.

**Confidence:** Medium

### [M25-ESPS] PLANNING — eye geometry implementation strategy

**Would have asked:** Should new eye shapes (oval, slit, square) be achieved by scaling existing `create_sphere` calls (scale non-uniformly to elongate or flatten) or by replacing the sphere with different Blender primitive calls?

**Assumption made:** Eye shapes are implemented by varying the `scale` tuple of `create_sphere` (circle = uniform, oval = X-elongated ellipsoid, slit = Y-narrow/Z-elongated, square ≈ cube-like ratios achieved via `create_box`). `circle` keeps the existing sphere. `oval` scales Y larger than Z. `slit` scales Z larger, Y narrow. `square` uses `create_box` from `blender_utils`. This avoids adding new Blender primitive wrappers and is consistent with how zone extras already shape geometry.

**Confidence:** Medium

### [M25-ESPS] PLANNING — pupil mesh approach

**Would have asked:** Should a pupil be a separate mesh object placed on the eye surface, or a material-only effect (UV mapping, shader)?

**Assumption made:** Pupil is a separate small mesh (sphere or flat disc via `create_sphere` with small Z scale = flat; a flat cylinder via `create_cylinder` thin-scale for slit; `create_box` for diamond approximation) placed on the front-facing surface of each eye sphere. This is consistent with how zone extras and the existing eye-on-head-sphere pattern work. The pupil receives its own material zone (no new color controls per ticket scope — the existing `head` or `extra` material zone is reused).

**Confidence:** Medium

### [M25-ESPS] PLANNING — which enemies get eye_shape/pupil controls

**Would have asked:** Should `eye_shape` and `pupil_*` controls be universal (all slugs) or per-slug?

**Assumption made:** Conservative: Controls apply to ALL animated enemy slugs (since all have heads with implied eyes, even if some are stylized spheres only). The Python builder reads the controls and applies shaping where eye mesh objects exist; if a slug has no explicit eye parts, the controls are declared but have no geometric effect (a future ticket can wire them). This is analogous to how `extra_zone_*` controls are declared per zone even when `kind=none`. Spec Agent should confirm scope during specification.

**Confidence:** Low — this is the most ambiguous design decision in the ticket.

### [M25-ESPS] PLANNING — frontend tab placement

**Would have asked:** Should eye shape/pupil controls appear on the Build tab (alongside `eye_count` for spider, `peripheral_eyes` for claw_crawler) or in a new dedicated Eyes tab?

**Assumption made:** Controls appear on the existing Build tab (same pattern as `eye_count`, `eye_distribution`, etc.). No new tab is needed. The controls are emitted as `AnimatedBuildControlDef[]` entries in `_ANIMATED_BUILD_CONTROLS` (or a new helper analogous to `_spider_eye_control_defs`).

**Confidence:** High

### [M25-ESPS] PLANNING — serialization key names

**Would have asked:** What key names to use in build_options JSON for the new controls?

**Assumption made:** Keys follow existing conventions:
- `eye_shape`: `select_str`, options `["circle", "oval", "slit", "square"]`, default `"circle"`
- `pupil_enabled`: `bool`, default `false`
- `pupil_shape`: `select_str`, options `["dot", "slit", "diamond"]`, default `"dot"`

These keys are validated in `_coerce_and_validate` / `options_for_enemy` following the existing `select_str` and `bool` patterns. Spec Agent should formalize key names.

**Confidence:** High
