### [M25-ESPS] SPECIFICATION — slug eye part ordering with pupils in spider

**Would have asked:** Should pupil parts be interleaved immediately after each eye in the `parts` list (eye0, pupil0, eye1, pupil1, ...) or grouped after all eyes (eye0, eye1, ..., pupil0, pupil1, ...)?

**Assumption made:** Grouped — all eyes appended first, then all pupils. This preserves the contiguous eye index range that `apply_themed_materials` currently uses (`parts[2..2+ec-1]`), extending it to `parts[2+ec..2+ec+ec-1]` for pupils. Interleaved ordering would require restructuring the existing material loop index arithmetic, increasing regression risk.

**Confidence:** High

### [M25-ESPS] SPECIFICATION — pupil material zone assignment

**Would have asked:** Should pupils receive the `extra` material (same as eye iris) or the `head` material?

**Assumption made:** `head` material, to create deliberate contrast between eye iris color (extra slot) and pupil (head slot). This makes the pupil visually distinct from the iris without adding a new material zone.

**Confidence:** Medium

### [M25-ESPS] SPECIFICATION — `slit` eye orientation (vertical vs horizontal)

**Would have asked:** Should `slit` eye shape be a vertical slit (narrow Y, tall Z from front view) or a horizontal slit (narrow Z, wide Y from front view)?

**Assumption made:** Vertical slit (narrow Y axis) — produces a cat/reptile-style vertical eye when viewed from the front (+X direction). This is the more recognizable "enemy eye slit" archetype. If incorrect, the implementation agent must flag before implementing.

**Confidence:** Medium

### [M25-ESPS] SPECIFICATION — `create_eye_mesh` and `create_pupil_mesh` placement

**Would have asked:** Should the two new helpers live in `blender_utils.py` (alongside `create_sphere`, `create_box`) or in a new `eye_geometry.py` module?

### [M25-ESPS] SPECIFICATION — slug part layout change from mod-2 to mod-3 pattern

**Would have asked:** Is it acceptable to break the slug builder's `i % 2` material assignment pattern when pupils are added?

**Assumption made:** Yes — the `% 2` pattern is a convenience that will no longer hold once a third part type (pupil) is inserted per eye+stalk group. The spec mandates explicit counting instead of modulo arithmetic when `pupil_enabled` is True. The implementation agent must update `apply_themed_materials` in `animated_slug.py` accordingly.

**Confidence:** High
