### [M5-LJEC] SPECIFICATION — allowed tokens for `_mesh_str` validation

**Would have asked:** The planning checkpoint assumed `_mesh_str` validates against a `ClassVar[tuple[str, ...]]`. What is the concrete attribute name for this ClassVar on `HumanoidSimpleRig`, and must it be defined per-ClassVar-field (one tuple per shape type), or one shared tuple for all string tuners?
**Assumption made:** Each string tuner (e.g. `ARM_END_SHAPE`, `LEG_END_SHAPE`) is a `ClassVar[str]`. Allowed tokens are defined as a single module-level constant in `limb_chain.py` (or `humanoid_simple.py`): `ALLOWED_END_SHAPES: Final[tuple[str, ...]] = ("none", "sphere", "box")`. `_mesh_str` accepts a `name` (the ClassVar key) and an `allowed` parameter (defaults to `ALLOWED_END_SHAPES`). This keeps validation co-located with the data and avoids per-field allowed-set ClassVars.
**Confidence:** Medium

**Would have asked:** If `build_options["mesh"]` contains a key with an unrecognized (invalid) token string, should `_mesh_str` raise `ValueError` immediately, or fall back to the class default silently?

### [M5-LJEC] SPECIFICATION — backward compatibility: `rig_from_bone_map` vs. direct construction

**Would have asked:** The planning note says: "build `RigDefinition.bones` directly by concatenating chain outputs, NOT via `rig_from_bone_map` dict (which loses ordering guarantees for multi-segment)." For the n=1 case, `limb_chain` would produce one `BoneSpec` per limb. Can we use `rig_from_bone_map` for the 3 non-limb bones (root, spine, head) and concatenate the chain outputs directly?
**Assumption made:** Yes. For backward compatibility with n=1, the implementation may build `RigDefinition.bones` by: (1) constructing non-limb bones directly as `BoneSpec` objects (root, spine, head), (2) concatenating `limb_chain(...)` output lists for all 4 limbs. The resulting `RigDefinition.bones` tuple must contain the same bones in the same order as the current `rig_from_bone_map` output when `ARM_SEGMENTS=1`, `LEG_SEGMENTS=1`. This is the backward-compatibility contract.
**Confidence:** High
