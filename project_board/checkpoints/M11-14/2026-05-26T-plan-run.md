## Discovery Summary

### Codebase state

- `scripts/enemies/enemy_base.gd` is a 32-line shell: `class_name EnemyBase extends CharacterBody3D`, identity exports (`enemy_id`, `enemy_family`, `mutation_drop`), `State` enum (NORMAL, WEAKENED, INFECTED), `current_state`, `set_base_state()`, `get_base_state()`, `get_esm()`.
- **No damage/health methods exist.** No `take_damage()`, `apply_poison()`, `apply_acid()`, `apply_slowness()` on EnemyBase.
- `scripts/attacks/attack_executor.gd` (COMPLETE, M11-05) calls `take_damage(damage, knockback)`, `apply_poison(duration, dps)`, `apply_acid(duration, dps)`, `apply_slowness(multiplier, duration)` via duck-typed `has_method()` guards. Until EnemyBase implements these, the attack pipeline has no runtime effect.
- `scripts/attacks/player_projectile_3d.gd` (COMPLETE, M11-05) also calls the same methods via duck typing on `_on_body_entered()`.
- `scripts/enemies/enemy_ai_controller.gd` (~232 lines) handles NORMAL→WEAKENED→INFECTED state transitions, patrol/chase AI, movement via `enemy_base.velocity` and `move_and_slide()`. Already reads `get_base_state()` and calls `set_base_state()`.
- Dependency tickets all in `done/`: M11-04 (AttackResource), M11-05 (AttackExecutor), M11-13 (Verify Damage Knockback).
- Existing test patterns: GDScript tests extend `res://tests/utils/test_utils.gd`, use inner mock classes, assertion helpers (`_assert_true`, `_assert_eq_float`, etc.), `run_all()` runner.

### Key design considerations for Spec Agent

1. **EnemyBase line budget**: Ticket requires ≤200 lines. With HP, take_damage(), damaged/died signals, death state, knockback impulse, WEAKENED threshold, plus existing 32 lines, the core fits ~80-100 lines. DoT (poison + acid, tick tracking, non-stacking refresh) and slowness (speed multiplier, duration timer) add ~60-80 lines. A helper extraction (e.g., `EnemyEffectTracker`) keeps EnemyBase lean and focused.
2. **Knockback impulse**: Must use `CharacterBody3D.velocity` in `_physics_process()`. Needs decay logic (e.g., exponential decay or linear decay over time). Must not permanently alter velocity — AI controller already sets velocity for patrol/chase.
3. **DoT tick interval**: 0.5s per AC. Two DoT types (poison, acid) with same tick rate but visually distinct. Non-stacking = re-application refreshes duration, does not add a second instance.
4. **Slowness modifier**: Speed multiplier applied to AI controller's movement speed. Duration-based. EnemyAIController reads movement constants — the modifier must be queryable (e.g., `get_speed_multiplier() -> float`).
5. **WEAKENED threshold**: Triggered when `current_hp` drops below 50% of `max_hp`. Must coexist with existing `set_base_state(WEAKENED)` — "whichever comes first" per AC. Once WEAKENED, should not re-trigger on subsequent damage.
6. **Death state**: When `current_hp` <= 0, disable AI, emit `died` signal. Whether to queue_free or play death animation is a presentation concern — spec should define the minimal contract.
7. **Integration with EnemyAIController**: AI controller currently sets `enemy_base.velocity` directly. Knockback impulse must not fight with AI movement. Options: knockback adds to velocity and decays, or knockback temporarily overrides AI movement.

### Assumptions logged

### [M11-14] PLANNING — Helper extraction strategy
**Would have asked:** Should DoT/modifier tracking be a child Node (composable, uses timers) or a plain RefCounted helper (lighter, manually ticked)?
**Assumption made:** Use a child Node (`EnemyEffectTracker`) — it can own its own `_process()` or `_physics_process()` for tick timing, which is cleaner than manual delta accumulation in EnemyBase. This also follows the codebase pattern of behavior composition via child nodes (like EnemyAIController is a child Node3D).
**Confidence:** High

### [M11-14] PLANNING — Knockback decay model
**Would have asked:** Should knockback decay be linear (constant rate) or exponential (friction-like)?
**Assumption made:** Use exponential decay (multiply by a decay factor each physics frame) as it produces smoother, more natural-feeling knockback that avoids an abrupt stop. The decay factor should be a tuning constant on EnemyBase.
**Confidence:** Medium

### [M11-14] PLANNING — Death behavior scope
**Would have asked:** Should death queue_free the enemy, play a death animation, or just disable AI and emit a signal?
**Assumption made:** Minimal death contract: disable AI controller, set a DEAD flag (or repurpose state), emit `died` signal. Do NOT queue_free — that's a presentation/lifecycle concern for a future ticket. The spec should define the death signal and the observable state change.
**Confidence:** High

### [M11-14] PLANNING — Slowness interaction with AI controller
**Would have asked:** How does the slowness modifier reach the AI controller's speed calculations?
**Assumption made:** EnemyBase exposes a `get_speed_multiplier() -> float` method. EnemyAIController calls it when computing chase/patrol speed. When no slowness is active, returns 1.0. The effect tracker manages the active slowness state and duration.
**Confidence:** High

## Risk Register

1. **Knockback vs AI velocity conflict**: EnemyAIController writes to `enemy_base.velocity` every `_physics_process`. Knockback impulse also writes velocity. Resolution: knockback adds a `_knockback_velocity` component that EnemyBase applies in its own `_physics_process` before `move_and_slide()`, separate from AI velocity. Spec Agent must define this interaction.
2. **DoT timing determinism in tests**: Real `_process` delta is non-deterministic. Tests must simulate time steps (call `_physics_process(0.5)` directly or use a testable delta-injection pattern).
3. **EnemyBase line budget**: With all features, the class could exceed 200 lines. Helper extraction to `EnemyEffectTracker` is the mitigation — must be planned in spec.
4. **WEAKENED state idempotency**: If `set_base_state(WEAKENED)` is called manually before HP drops below 50%, the HP threshold should not re-trigger. Spec must clarify one-way latch semantics.
5. **Death state and queue_free**: Death disables AI but does NOT queue_free. If other systems (e.g., run scene assembler) expect enemies to be freed, there could be orphaned nodes. Scoped to "emit `died` signal" — cleanup is a presentation/lifecycle concern for a future ticket.
