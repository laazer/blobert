## Assumptions Logged

### [M11-14] SPECIFICATION — Knockback overwrite semantics
**Would have asked:** Should multiple rapid knockback hits stack additively or should the latest knockback overwrite the previous?
**Assumption made:** Latest knockback overwrites previous (not additive). This is the simplest model and avoids explosive velocity accumulation from rapid multi-hit attacks.
**Confidence:** High

### [M11-14] SPECIFICATION — DoT damage signal semantics
**Would have asked:** Should DoT tick damage emit the `damaged` signal or a separate signal?
**Assumption made:** DoT ticks emit a separate `dot_tick(effect_name, tick_damage, current_hp)` signal. The `damaged` signal is reserved for direct hits (take_damage calls). This allows UI systems to differentiate between hit feedback and ongoing DoT indicators.
**Confidence:** High

### [M11-14] SPECIFICATION — move_and_slide() ownership
**Would have asked:** Since EnemyBase now has _physics_process for knockback, should it also own the move_and_slide() call (removing it from EnemyAIController)?
**Assumption made:** Yes. EnemyBase._physics_process() is the sole caller of move_and_slide(). EnemyAIController sets velocity components but does NOT call move_and_slide(). This prevents double-physics-step bugs. The AI controller processes first (child before parent), then EnemyBase applies knockback and calls move_and_slide().
**Confidence:** High
