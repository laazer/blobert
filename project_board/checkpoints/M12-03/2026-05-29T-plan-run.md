## Key Discovery: The Fusion State Ambiguity

This is the most important finding from planning and must be surfaced before the Spec Agent begins.

### Two distinct "fusion" concepts coexist in this codebase

**Concept A — Speed-boost fusion (`_fusion_active` / `is_fusion_active()`):**
- Set by `FusionResolver.resolve_fusion()` calling `player.apply_fusion_effect(duration, multiplier)`
- At the moment of resolution, `consume_fusion_slots()` is also called, clearing both mutation slots
- While `_fusion_active == true`: both mutation slots are EMPTY
- `_fusion_timer` counts down; when it hits 0, `_fusion_active = false`
- This is the speed-boost "you just fused" state

**Concept B — Dual mutation state (`a_filled and b_filled`):**
- Both mutation slots contain active mutations, but the player has NOT yet triggered fusion
- `_mutation_slot.get_slot(0).is_filled() == true` AND `_mutation_slot.get_slot(1).is_filled() == true`
- This is the pre-fusion state: player has two mutations and can attack with the fused attack
- OR has triggered fusion already (slots consumed), in which case both are empty

**The current `_try_attack()` implementation (lines 445-482 in `player_controller_3d.gd`):**
- Routes to fused attack when `a_filled and b_filled` (Concept B)
- Does NOT use `is_fusion_active()` at all in the routing decision
- This was fully specified and tested in M12-01 as COMPLETE

**The ticket's AC says:**
- "When fusion is active, `attack` input routes to the fusion attack for that combination"
- "`is_fusion_active()` on PlayerController3D is used to determine routing (no duplicate state)"

This creates an apparent conflict: `is_fusion_active()` is true only when slots are EMPTY (post-fusion
speed boost). At that moment, there are no slot IDs to look up in AttackDatabase.

The Spec Agent MUST freeze which semantic is intended:

- **Interpretation A:** "Fusion is active" = speed-boost timer running (slots consumed). This would
  require storing the fused combo identity before consuming slots, and routing to that stored attack
  during the speed-boost window. This is a NEW feature requiring implementation.

- **Interpretation B:** "Fusion is active" = both mutation slots are filled (dual mutation state,
  pre-fusion trigger). This is exactly what M12-01 implemented. The AC wording is misleading but
  the intent is the existing behavior. `is_fusion_active()` would need to be confirmed as NOT
  the routing signal — the slot-fill check is the routing signal. No implementation needed.

- **Interpretation C:** M12-03 is a hybrid — the "fusion framework" refers to the dual-mutation
  routing (M12-01 AC is "integration of M9 framework"), and M12-03 adds a separate dimension:
  during the speed-boost window, the fused attack should still fire (using the last known combo).
  This would require tracking the last fused combo ID before slot consumption.

The conservative assumption is Interpretation B (existing behavior, no new implementation). However
the ticket's explicit reference to `is_fusion_active()` as the routing signal is too specific to
dismiss — the Spec Agent must read the existing code and freeze the correct interpretation.
