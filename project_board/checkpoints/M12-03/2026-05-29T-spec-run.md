## Central Design Decision: FAF-DD-1 (Frozen)

### [M12-03] SPECIFICATION — is_fusion_active() vs both-slots-filled routing gate
**Would have asked:** The ticket AC says "`is_fusion_active()` on PlayerController3D is used to
determine routing." But code shows `_try_attack()` (lines 445-482) has zero references to
`is_fusion_active()` or `_fusion_active`. The routing gate is `if a_filled and b_filled:` (line 464).
Which is correct?

**Assumption made:** Interpretation B is frozen as correct. The AC's `is_fusion_active()` reference
is a semantic mismatch: the AC means "the player is in a state where fusion attack is possible"
(both slots filled), NOT the `_fusion_active` speed-boost timer. Code evidence:
- `player_controller_3d.gd` line 464: `if a_filled and b_filled:` — this is the fused routing gate
- `player_controller_3d.gd` line 603: `func is_fusion_active() -> bool: return _fusion_active` — this
  is the speed-boost timer only; temporally exclusive from "both slots filled"
- `fusion_resolver.gd` lines 51-64: `resolve_fusion()` calls `apply_fusion_effect()` then
  `consume_fusion_slots()`. Post-resolve: `is_fusion_active() == true` but BOTH slots are EMPTY.
  In that state `_try_attack()` returns early at line 457 (neither slot filled = no attack).

The "no duplicate state" constraint in AC-4 is satisfied because the routing derives from a single
authoritative source: slot fill state. Consulting `is_fusion_active()` inside `_try_attack()` would
ADD duplicate state, not remove it.

**Confidence:** High — verified directly from code at three locations.

### [M12-03] SPECIFICATION — Speed-boost window attack behavior
**Would have asked:** Should the fused attack be fireable during the speed-boost timer window
(post-resolve_fusion, slots consumed)?

**Assumption made:** No. This is explicitly out of scope for M12-03. During the speed-boost window,
`_try_attack()` correctly returns early because both slots are empty. If future game design requires
attack dispatch during the speed-boost window, that requires a separate ticket with new state (storing
last combo identity before slot consumption) and a new routing branch. Deferred scope documented in
spec Section 9.

**Confidence:** High — confirmed by code logic and ticket scope boundary.
