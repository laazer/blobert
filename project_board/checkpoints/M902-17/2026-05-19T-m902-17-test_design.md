## Decisions Made (Checkpoint Protocol)

### Decision 1: Mock-Only Testing vs. Real Gate Execution
**Would have asked:** Should behavioral tests call real gate implementations or use mocks?

**Assumption made:** Use realistic mock gate outputs (M902-01 schema-compliant) for behavioral tests. Real gate execution is tested in Task 4 (Integration phase) where gate_runner.py is invoked.

**Confidence:** HIGH  
**Rationale:** Behavioral tests should validate pipeline logic, not gate implementations. Real gates are tested separately; mocks allow fast deterministic tests (<30s execution target).

### Decision 2: Scope of "Behavioral vs. Adversarial"
**Would have asked:** Where's the line between behavioral and adversarial tests?

**Assumption made:** This file contains behavioral tests (happy-path, early-exit, basic failures, edge cases). Adversarial tests (missing inputs, malformed JSON, schema violations, module loading failures) are a separate test file (Task 3, Test Breaker).

**Confidence:** HIGH  
**Rationale:** Specification defines separate test suites; behavioral suite = 12+ tests; adversarial suite = 15+ tests. Clear separation prevents test file bloat.

### Decision 3: Test Organization & Naming
**Would have asked:** How to organize 38 tests for readability and traceability?

**Assumption made:** Organize by stage/component (TestStage0Classification, TestGateRegistry, etc.). Test names follow pattern `test_<stage>_<scenario>_<outcome>` or `test_<component>_<case>`. All tests are methods within classes for better IDE navigation.

**Confidence:** HIGH  
**Rationale:** Matches project conventions (other test files use class-based organization); names are self-documenting and map to AC language.

### Decision 4: Fixture Scope & Reusability
**Would have asked:** Should fixtures be in conftest.py or in the test file?

**Assumption made:** All fixtures defined in this file (test_m902_17_pipeline_validation.py) since they're specific to M902-17 validation. Shared fixtures (repo_root, gate_runner, etc.) remain in conftest.py.

**Confidence:** HIGH  
**Rationale:** M902-17 fixtures are not reused by other tests; keeping them co-located simplifies fixture understanding and maintenance. Tests can discover repo paths via pytest's standard repo_root fixture if needed.

### Decision 5: M902-01 Schema Validation Logic
**Would have asked:** Should schema validation be inline or a helper method?

**Assumption made:** Implemented as a static method `_validate_gate_output()` in TestSchemaCompliance class. This method checks for required fields (status, violations, remediation_hints, metadata) and validates their types and contents.

**Confidence:** HIGH  
**Rationale:** Reusable across multiple test methods; clear contract for what "schema compliant" means; can be extracted to conftest.py later if needed by other tests.
