## Decisions Made (Checkpoint Protocol)

### Decision 1: Adversarial vs. Integration Testing Scope
**Would have asked:** Should adversarial tests call real gates or use mocks?

**Assumption made:** Adversarial tests use mocks and directly validate assumptions about schema, registry, and routing logic. Real gate execution is tested in Task 4 (Integration phase). Adversarial tests focus on failure modes and edge cases that would expose implementation bugs.

**Confidence:** HIGH
**Rationale:** Adversarial testing targets implementation correctness assumptions, not runtime behavior. Real gates have their own tests; adversarial suite validates framework-level contracts.

### Decision 2: Schema Validation vs. Gate Module Validation
**Would have asked:** Should adversarial tests focus on schema or module loading?

**Assumption made:** Both. Schema violation tests (5) focus on output structure. Module loading tests (3) focus on gate discovery and callable validation. Both are prerequisites for a working pipeline.

**Confidence:** HIGH
**Rationale:** Schema violations would break downstream stages silently; module loading failures would break pipeline execution. Both are critical error conditions.

### Decision 3: Boundary Condition Thresholds
**Would have asked:** How precisely should off-by-one errors be tested?

**Assumption made:** Explicit boundary tests:
- risk=2 should skip Stages 5–6; risk=3 should include (boundary at 3)
- risk=5 should be WARN; risk=6 should be ESCALATE (boundary at 6)
- suppression count=2 is allowed; count=3+ escalates (threshold at 3)

**Confidence:** HIGH
**Rationale:** Off-by-one errors are common in threshold logic. Explicit boundary testing catches these bugs.

### Decision 4: Performance Test Approach
**Would have asked:** Should performance tests measure real or mocked gates?

**Assumption made:** Mocked gates for adversarial suite (validates orchestration overhead). Real gates tested in Task 4 (Integration phase). Performance tests verify that framework doesn't add significant overhead (orchestration should be <100ms for 8 stages).

**Confidence:** HIGH
**Rationale:** Adversarial suite needs to be fast (<45s total); real gate timing variability would make adversarial tests flaky. Real performance metrics come from integration tests with actual gate implementations.
