## Planner Agent Run — 2026-03-30

### Decision: Headless .tscn Generation is Feasible (Option 1 chosen)

**Analysis of load_assets.gd:**

The script uses only these APIs:
- `DirAccess.open()`, `DirAccess.make_dir_recursive_absolute()` — headless-compatible
- `load(glb_path)` / `PackedScene.instantiate()` — works headlessly after `godot --import` populates the .import cache; the 12 GLBs already have .import files
- `CharacterBody3D.new()`, `Node3D.new()`, `CollisionShape3D.new()`, `Area3D.new()`, `Marker3D.new()`, `VisibleOnScreenNotifier3D.new()` — all node constructors work headlessly
- `mesh.get_aabb()` — works without a render server
- `ResourceSaver.save(packed, path)` — works headlessly for .tscn files
- `EnemyNameUtils.extract_family_name()` — pure string logic, no editor dependency

The only blocker is `@tool extends EditorScript` at the top. Removing those two declarations and replacing `func _run()` with an `extends SceneTree` / `func _init()` entry point makes the script fully runnable via `timeout 300 godot -s <script>`.

**Decision:** Write `scripts/asset_generation/generate_enemy_scenes.gd` as a new headless tool script (not modifying load_assets.gd, which must remain for editor use). The new script duplicates the generation logic and is the automatable entry point for this pipeline.

**GLB naming note:** The 12 GLBs use the `_animated_` infix (e.g., `adhesion_bug_animated_00.glb`). `EnemyNameUtils.extract_family_name()` already strips numeric suffixes and `animated` tokens, so `adhesion_bug_animated_00` → `adhesion_bug`. The MUTATION_BY_FAMILY map in load_assets.gd contains all 4 families. No changes needed to EnemyNameUtils.

**Level placement:** The existing test_movement_3d.tscn already has two `enemy_infection_3d.tscn` instances. The plan will replace those with 4 instances of the newly generated family .tscn files, one per family. The generated scenes use `EnemyBase` (CharacterBody3D + enemy_id/enemy_family/mutation_drop exports) but NOT `EnemyInfection3D`. To satisfy the acceptance criteria (weakening + infection + mutation drop), each placed enemy must include an `EnemyInfection3D` node. The plan adds this as a child of each generated enemy root or creates wrapper scenes.

**Wrapper scene approach:** Rather than modifying the generated .tscn directly (which would be overwritten on regen), the Spec Agent will design thin "placement wrapper" scenes under `scenes/enemies/placed/` that instance the generated .tscn and add `EnemyInfection3D` + collision layer configuration.

**Assumption:** `global_transform` on MeshInstance3D nodes inside instantiated GLBs may be identity (not yet placed in the scene tree), so AABB computation via `global_transform * corner` could be world-space zero. Conservative fallback: generate a unit BoxShape3D if AABB is zero. The Spec Agent must confirm this in the headless generation script.

## Spec Agent Run — 2026-03-30

### Decision: Wrapper Scene Architecture Confirmed

**Analysis of EnemyInfection3D:**

`EnemyInfection3D` extends `BasePhysicsEntity3D` which extends `CharacterBody3D`. It is NOT a child node — it IS a CharacterBody3D root itself. It cannot be meaningfully added as a child of another CharacterBody3D root (generated enemy), because:
1. Nesting CharacterBody3D under CharacterBody3D is architecturally unsupported in Godot physics.
2. EnemyInfection3D._ready() walks `get_parent()` looking for `InfectionInteractionHandler` — this requires it to be a direct or close scene-tree child of the level root.
3. EnemyInfection3D owns its own CollisionShape3D (BodyShape) and InteractionArea.

**Conclusion:** The generated .tscn (EnemyBase root) and EnemyInfection3D CANNOT be composed into a single root hierarchy. The acceptance criterion "each enemy can be weakened and infected via existing infection loop" requires that a placed enemy IS an EnemyInfection3D (or structurally equivalent). Two viable approaches:

**Option A (Chosen):** Placement wrappers do NOT wrap the generated .tscn. Instead, each placed enemy IS an `enemy_infection_3d.tscn` instance — identical to the existing Enemy/Enemy2 nodes — with `enemy_id`, `enemy_family`, and `mutation_drop` set as overrides on the instance in the level .tscn. The generated .tscn files serve as visual/asset reference; their EnemyBase exports are the canonical source of truth for the metadata values. The level cares about infection, not about which visual model.

**Option B (Future):** Replace EnemyInfection3D's gobot GLB with the generated GLB per family. This is a deeper visual integration and is out of scope for this ticket.

**Consequence for spec:**
- Placement wrapper scenes under `scenes/enemies/placed/` are NOT needed for this ticket. The level will instance `enemy_infection_3d.tscn` four times (same scene, different names and positions), with per-instance property overrides for `enemy_id`, `enemy_family`, and `mutation_drop`.
- The generated .tscn files are still produced and committed (they are the spec output for sub-contracts 1 and 2), and tests verify their structure. They are not yet used in the level.
- Sub-contract 3 (placement wrappers) is redefined: no new wrapper .tscn files are created. Instead, the level scene directly uses `enemy_infection_3d.tscn` with instance property overrides.

**Decision on existing Enemy/Enemy2 nodes:**
`test_3d_scene.gd` has NO references to nodes named "Enemy" or "Enemy2". No other test in the test suite references nodes by those names in `test_movement_3d.tscn` specifically. The existing Enemy and Enemy2 nodes can be removed and replaced with 4 new family-named instances.

**AABB computation in headless mode:**
When GLB scenes are instantiated via `PackedScene.instantiate()` but NOT added to a SceneTree, `mesh_instance.global_transform` is identity (Transform3D.IDENTITY). The AABB computed from `global_transform * corner` is therefore in local space (identical to the raw mesh AABB). This is correct for a fresh root CharacterBody3D. The spec explicitly documents that `global_transform` will be identity in headless mode and this is the intended behavior — the resulting shape is in the root's local space, which is what the physics engine needs for a CollisionShape3D child.

**Fallback condition:** If `_compute_combined_aabb()` returns AABB with `size == Vector3.ZERO` (no MeshInstance3D found or all meshes have zero-sized bounds), the generator MUST emit a unit BoxShape3D with `size = Vector3(1.0, 1.0, 1.0)` (half-extents 0.5 on each axis). This matches the ticket instruction "use unit BoxShape3D (0.5, 0.5, 0.5)" — note: BoxShape3D.size is full extent, so Vector3(1.0, 1.0, 1.0) gives half-extents of 0.5.

**Test runner registration:**
`run_tests.gd` uses automatic discovery: it recursively collects all `.gd` files under `res://tests/` whose filename starts with `test_`. No manual registration is required. The two new test files placed in `tests/scenes/enemies/` will be discovered automatically. The `_LOGGING_TESTS` preload in run_tests.gd is a one-off for parse-order reasons and does not establish a pattern that new tests must follow.
