## Checkpoint Assumptions

### [M12-02] PLANNING — Design: code registration vs .tres files
**Would have asked:** Should the 6 fused attacks be created as Godot `.tres` resource files in `res://attacks/fused/`, as the ticket title states, OR as named constants + `register_fused_attack()` calls inside `_register_defaults()`, consistent with how all 4 base attacks are registered?
**Assumption made:** Code registration inside `_register_defaults()` is the correct approach. Rationale: the entire existing attack system (all 4 base attacks) uses code registration, never `.tres` files. There is no `attacks/` directory anywhere in the project root. Introducing `.tres` files for fused attacks only would create an asymmetry with no stated rationale. The ticket notes themselves say "Can be created as .tres files manually or programmatically after initial data is defined", which acknowledges the code path is valid. The conservative, convention-matching choice is code registration. The Spec Agent must validate this assumption and document it as a Design Decision in the spec before TEST_DESIGN begins.
**Confidence:** High

## Key Design Decision Logged

**Choosing code registration over `.tres` files.** The ticket title says "Create fused attack resource files (.tres)" but the entire attack system — all 4 base attacks — is code-registered in `_register_defaults()`. There is no `attacks/` directory in the project. Creating `.tres` files would be novel infrastructure with no prior art in this codebase and would require Godot resource file parsing at load time. The implementation approach selected is: add 6 named-constant blocks to `attack_database.gd` and 6 `register_fused_attack()` calls at the end of `_register_defaults()`. This matches existing conventions exactly. The ticket ACs are satisfied either way (AttackDatabase can load all fused attacks, tests validate them); the code-registration approach is strictly simpler and consistent.
