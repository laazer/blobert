# TICKET: animation_controller_script

Title: Implement shared enemy animation controller script

## Description

Create `scripts/enemies/enemy_animation_controller.gd` that reads the current EnemyStateMachine state each frame and plays the matching animation clip on the enemy's AnimationPlayer. The controller must handle state transitions without animation pop (crossfade where applicable).

States to animation mapping:
- NORMAL + not moving → `Idle`
- NORMAL + moving → `Walk`
- WEAKENED → `Idle` (slower playback speed)
- INFECTED → `Idle` (frozen or minimal)
- On hit received → `Hit` (one-shot, then return to state clip)
- On death → `Death` (one-shot, no loop, holds last frame)

## Acceptance Criteria

- Script exists at `scripts/enemies/enemy_animation_controller.gd`
- Attaches to generated enemy scenes as a child node of the enemy root
- Plays `Idle` when enemy is stationary in NORMAL state
- Plays `Walk` when enemy is moving
- Plays `Hit` as a one-shot on damage received, then resumes state clip
- Plays `Death` on death; holds last frame; no loop
- WEAKENED state plays `Idle` at 0.5× speed
- No animation pop on state transition (use `AnimationPlayer.play` with blend time ≥ 0.1s)
- `run_tests.sh` exits 0

## Dependencies

- `blender_animation_export` — animation clips must exist in GLBs
- EnemyStateMachine must expose current state readably

## Specification

### Requirement ACS-1: Script Identity and Node Attachment

#### 1. Spec Summary

- **Description:** A new GDScript file `scripts/enemies/enemy_animation_controller.gd` defines a class named `EnemyAnimationController` that extends `Node`. It is attached as a direct child of each generated enemy root (`CharacterBody3D` with `EnemyBase` script) produced by `generate_enemy_scenes.gd`. The `generate_enemy_scenes.gd` script must be updated to instantiate and attach one `EnemyAnimationController` node to each generated enemy scene.
- **Constraints:** The script must extend `Node` (not `Node3D`, not `RefCounted`). It must be placed exactly at `res://scripts/enemies/enemy_animation_controller.gd`. The node is a direct child of the enemy root — not nested inside `Visual` or any other intermediate node.
- **Assumptions:** The `generate_enemy_scenes.gd` update is in scope for the same ticket as this script. No other ticket handles this wiring.
- **Scope:** Applies to all enemy scenes generated by `generate_enemy_scenes.gd`. Does not apply to manually authored enemy scenes (none exist at time of writing).

#### 2. Acceptance Criteria

- AC-1.1: File `res://scripts/enemies/enemy_animation_controller.gd` exists and parses without error (Godot script load succeeds).
- AC-1.2: The class is declared as `class_name EnemyAnimationController` and `extends Node`.
- AC-1.3: `generate_enemy_scenes.gd` instantiates `EnemyAnimationController` and adds it as a direct child of the `CharacterBody3D` root before packing the scene. The node's `owner` is set to `root` so it is saved into the `.tscn` file.
- AC-1.4: In any generated `.tscn` file, the `EnemyAnimationController` node appears as a direct child of the scene root (depth 1), confirmed by scene inspection.

#### 3. Risk & Ambiguity Analysis

- Risk: `generate_enemy_scenes.gd` preloads the script by path. If the path is wrong the scene generator will push a warning and continue, silently producing scenes without the controller. The Implementation Agent must verify the preload path resolves.
- Edge case: If the script file does not exist when `generate_enemy_scenes.gd` runs, the preload call will fail at parse time. The scene generator must guard this gracefully or the spec must require the script file to exist before running the generator. The spec requires both to ship together.

#### 4. Clarifying Questions

- None. All resolved by assumptions above and in planning checkpoints.

### Requirement ACS-2: Variable API and Dependency Injection

#### 1. Spec Summary

- **Description:** `EnemyAnimationController` exposes two `@export` variables for primitive configuration and two plain `var` declarations for object references. The object references cannot be `@export` due to Godot 4 runtime constraints. No constructor arguments are used. The script must not crash if any reference is null — null checks occur in `_ready()`.

  **`animation_player` (plain var, no @export):** Resolved at runtime in `_ready()` via `get_parent().find_child("AnimationPlayer", true, false)`. If the result is non-null it is stored; if null a warning is emitted and `_ready_ok` remains false. Tests may inject a stub directly by assigning `controller.animation_player = stub` before `_ready()` runs — when a non-null value is already present, `find_child()` is skipped. Rationale: the AnimationPlayer is always a sibling in the generated scene tree; `@export` is not required and would impose a type annotation that Godot 4.6.1 enforces at runtime, breaking stub injection in headless tests.

  **`state_machine` (plain var, no @export):** Injected via `setup(esm: Object) -> void`. The generator calls `setup(null)` after instantiation (pre-M15 placeholder). Tests assign `controller.state_machine = stub` directly. Rationale: `EnemyStateMachine` extends `RefCounted`, not `Node`. Godot 4 cannot serialize a `RefCounted` instance through an `@export` in `.tscn` files, and a `Node`-typed export would reject assignment of a `RefCounted` instance at runtime.

- **Constraints:** `animation_player` and `state_machine` are declared as `var … : Object = null` (no `@export`). `move_threshold` and `blend_time` retain `@export` (primitive `float` types, correctly serializable). No `_init()` parameters. The script must not crash if any reference is left null.
- **Assumptions:** Godot 4.6.1 enforces `@export` type annotations at runtime when assigning via script in headless mode; this was confirmed by the Implementation Agent (see checkpoint run-2026-04-01-impl-fix.md). Using `Object` as the declared type is the accepted compromise for headless testability.
- **Scope:** Public API surface of `EnemyAnimationController`.

The variable declarations:

| Variable | Declaration | Default | Injection mechanism | Purpose |
|---|---|---|---|---|
| `animation_player` | `var animation_player: Object = null` | `null` | `_ready()` via `find_child()`, or direct assignment before `_ready()` | Reference to the AnimationPlayer node |
| `state_machine` | `var state_machine: Object = null` | `null` | `setup(esm: Object) -> void`, or direct assignment before `_ready()` | Reference to EnemyStateMachine |
| `move_threshold` | `@export var move_threshold: float` | `0.1` | Inspector / scene file | Velocity magnitude threshold above which enemy is considered moving |
| `blend_time` | `@export var blend_time: float` | `0.15` | Inspector / scene file | Crossfade duration in seconds for non-one-shot transitions |

#### 2. Acceptance Criteria

- AC-2.1: `animation_player` is declared as `var animation_player: Object = null` with no `@export` decorator. `state_machine` is declared as `var state_machine: Object = null` with no `@export` decorator. `move_threshold` is declared as `@export var move_threshold: float = 0.1`. `blend_time` is declared as `@export var blend_time: float = 0.15`.
- AC-2.2: `move_threshold` default is `0.1` (float).
- AC-2.3: `blend_time` default is `0.15` (float).
- AC-2.4: `animation_player` and `state_machine` default to `null`.
- AC-2.5: Assigning a non-null stub Object to `animation_player` or `state_machine` before `_ready()` runs does not cause a type error at runtime.
- AC-2.6: `setup(esm: Object) -> void` is a public method that stores `esm` into `state_machine`. Calling it with `null` is valid and results in `state_machine == null`.
- AC-2.7: When `animation_player` is null at the start of `_ready()`, the controller calls `get_parent().find_child("AnimationPlayer", true, false)` and stores the result. If the result is non-null, `animation_player` is set to that result. If the result is null, a warning is emitted.
- AC-2.8: When `animation_player` is already non-null at the start of `_ready()` (injected by a test), `find_child()` is NOT called — the injected value is used as-is.

#### 3. Risk & Ambiguity Analysis

- `state_machine` is typed as `Object` so that test stubs (RefCounted-based inner classes) can be assigned without type errors. The controller calls `state_machine.get_state()` via dynamic dispatch; GDScript duck-typing permits this.
- Rationale for no `@export` on `state_machine`: `EnemyStateMachine extends RefCounted`. Godot 4 cannot serialize a `RefCounted`-derived object through an `@export` in `.tscn` files. A `Node`-typed export would reject assignment of a `RefCounted` at runtime. The `setup()` injection pattern is the accepted solution.
- Rationale for no `@export` on `animation_player`: Godot 4.6.1 enforces `@export` type annotations at runtime in headless mode. An `@export var animation_player: AnimationPlayer` export causes "Invalid assignment" when a plain Object stub is assigned in tests, breaking all 39 tests. Runtime resolution via `find_child()` plus direct assignment for tests is the accepted solution.
- Risk: If a future Godot version changes `find_child()` behavior or the scene topology changes (AnimationPlayer no longer a direct sibling), `_ready()` resolution will silently fail and emit a warning, leaving `_ready_ok = false`. This is the correct defensive behavior.

#### 4. Clarifying Questions

- None. All resolved by implementation checkpoints (run-2026-04-01-impl-fix.md) and this spec update.

### Requirement ACS-3: Null-Guard Initialization

#### 1. Spec Summary

- **Description:** `_ready()` inspects both `animation_player` and `state_machine`. If either is null, the controller emits a warning via `push_warning()` and sets an internal flag that disables all `_physics_process` logic. The node remains in the scene tree but is effectively dormant.
- **Constraints:** Must not call `queue_free()` or remove itself. Must not crash. Warning text must include the node's name and which export is null.
- **Assumptions:** An enemy scene where the AnimationPlayer has not yet been added (pre-`blender_animation_export`) is a valid deployment scenario.
- **Scope:** `_ready()` method only.

#### 2. Acceptance Criteria

- AC-3.1: If `animation_player` is null when `_ready()` runs, `push_warning()` is called with a string containing `"animation_player"` and the node name.
- AC-3.2: If `state_machine` is null when `_ready()` runs, `push_warning()` is called with a string containing `"state_machine"` and the node name.
- AC-3.3: After either null-guard fires, `_physics_process()` executes zero calls on `animation_player` or `state_machine` for the lifetime of the node.
- AC-3.4: When both references are valid, no warning is emitted and `_physics_process()` runs normally.
- AC-3.5: The null-guard does not call `queue_free()`, `set_process(false)`, or any Node removal method — the flag-based dormancy approach is required (internal bool `_ready_ok: bool = false`).

#### 3. Risk & Ambiguity Analysis

- Using `set_process(false)` would be simpler but prevents future hot-reload scenarios where the export is assigned after `_ready`. The flag approach costs one branch per physics frame but is safer.
- Edge case: Both references null simultaneously — both warnings must fire, not just the first.

#### 4. Clarifying Questions

- None.

### Requirement ACS-4: State-to-Clip Resolution

#### 1. Spec Summary

- **Description:** Each `_physics_process` tick, the controller resolves the current animation target — a (clip_name: String, speed: float) tuple — from the EnemyStateMachine state string and the parent's velocity. The resolution rules are strict and exhaustive.

Resolution table (evaluated in priority order):

| Condition | Resolved clip | Resolved speed |
|---|---|---|
| `state_machine.get_state() == "dead"` | `"Death"` | `1.0` |
| `state_machine.get_state() == "infected"` | `"Idle"` | `0.0` |
| `state_machine.get_state() == "weakened"` | `"Idle"` | `0.5` |
| `state_machine.get_state()` is `"idle"` or `"active"` AND `get_parent().velocity.length() >= move_threshold` | `"Walk"` | `1.0` |
| `state_machine.get_state()` is `"idle"` or `"active"` AND `get_parent().velocity.length() < move_threshold` | `"Idle"` | `1.0` |
| Any other state string (future/unknown) | `"Idle"` | `1.0` (fallback) |

- **Constraints:** The clip name strings are exact string literals: `"Idle"`, `"Walk"`, `"Death"`. Casing is significant. The fallback for unknown states must not crash.
- **Assumptions:** `get_parent()` returns the enemy's `CharacterBody3D` root which has a `velocity: Vector3` property. This is guaranteed by the scene topology (ACS-1).
- **Scope:** Internal resolution logic, called from `_physics_process` and from the Hit resume path (ACS-6).

#### 2. Acceptance Criteria

- AC-4.1: State `"dead"` resolves to clip `"Death"`, speed `1.0`.
- AC-4.2: State `"infected"` resolves to clip `"Idle"`, speed `0.0`.
- AC-4.3: State `"weakened"` resolves to clip `"Idle"`, speed `0.5`.
- AC-4.4: State `"active"` with `velocity.length() >= 0.1` (default threshold) resolves to `"Walk"`, speed `1.0`.
- AC-4.5: State `"idle"` with `velocity.length() >= 0.1` resolves to `"Walk"`, speed `1.0`.
- AC-4.6: State `"active"` with `velocity.length() < 0.1` resolves to `"Idle"`, speed `1.0`.
- AC-4.7: State `"idle"` with `velocity.length() < 0.1` resolves to `"Idle"`, speed `1.0`.
- AC-4.8: An unknown state string (e.g., `"stunned"`) resolves to `"Idle"`, speed `1.0` without error.
- AC-4.9: `move_threshold` set to `0.5` causes velocity `0.3` to resolve to `"Idle"` (not `"Walk"`).

#### 3. Risk & Ambiguity Analysis

- `"dead"` is checked first — this is intentional priority. A dead enemy must never play Walk or Idle clips regardless of residual velocity on the CharacterBody3D.
- `"infected"` speed `0.0` means the AnimationPlayer plays the Idle clip but advances zero frames per second — effectively frozen on whatever frame the transition landed on. This is distinct from `stop()`. The Implementation Agent must not call `stop()` for infected; it must call `play("Idle", blend_time)` and then set `animation_player.speed_scale = 0.0`.
- Risk: `animation_player.speed_scale` is a property on `AnimationPlayer`, not a parameter to `play()`. The stub must expose this property.

#### 4. Clarifying Questions

- None.

### Requirement ACS-5: Transition Application (No-Pop Guarantee)

#### 1. Spec Summary

- **Description:** After resolving the target (clip_name, speed) tuple, the controller compares it to the currently active (clip_name, speed). If both match, no call to `animation_player.play()` is made (idempotent). If either differs, `animation_player.play(clip_name, blend_time)` is called and `animation_player.speed_scale` is set to `speed`. The `blend_time` export value is used for all non-one-shot transitions.
- **Constraints:** One-shot clips (`"Hit"`, `"Death"`) are handled separately (ACS-6, ACS-7) and do not go through this path.
- **Assumptions:** The stub AnimationPlayer's `play()` signature matches Godot 4's: `play(name: String = "", custom_blend: float = -1, custom_speed: float = 1.0, from_end: bool = false) -> void`. The spec uses positional args `play(clip_name, blend_time)`.
- **Scope:** `_physics_process` dispatch path, excluding one-shot states.

#### 2. Acceptance Criteria

- AC-5.1: When the resolved clip and speed are identical to the currently playing clip and speed, `animation_player.play()` is NOT called in that tick.
- AC-5.2: When the resolved clip differs from the currently playing clip, `animation_player.play(new_clip, blend_time)` is called exactly once.
- AC-5.3: When only speed differs (same clip name), `animation_player.play()` is called with `blend_time` and `animation_player.speed_scale` is updated.
- AC-5.4: `blend_time` value `0.15` is passed as the second argument to `play()` by default.
- AC-5.5: Changing `blend_time` export to `0.0` causes `play()` to be called with `0.0` as second argument.
- AC-5.6: The controller tracks the last-applied (clip_name, speed) in internal variables `_current_clip: String` and `_current_speed: float` (initial values `""` and `1.0` respectively).

#### 3. Risk & Ambiguity Analysis

- The idempotency check (AC-5.1) is critical for performance: without it, `play()` would be called every physics frame even when nothing changes.
- Risk: If `animation_player.speed_scale` is set without calling `play()` on the same frame (when only speed changes), the AnimationPlayer may need to be replayed to apply the new speed. The spec requires `play()` to be called whenever speed changes, not just when clip name changes.

#### 4. Clarifying Questions

- None.

### Requirement ACS-6: Hit One-Shot Animation

#### 1. Spec Summary

- **Description:** `EnemyAnimationController` exposes a public method `trigger_hit_animation() -> void`. When called, the controller saves the current state-resolved clip (if not already in a Hit) into `_prior_clip: String` and `_prior_speed: float`, then calls `animation_player.play("Hit", 0.0)` (blend time zero for immediate response). The controller enters "hit active" mode tracked by `_hit_active: bool`. Each `_physics_process` tick while `_hit_active` is true, the controller checks whether the Hit clip has finished playing (via `animation_player.is_playing()` returning false OR `animation_player.current_animation != "Hit"`). When Hit finishes, the controller re-evaluates the current state (full ACS-4 resolution) and transitions to that clip using `blend_time`. `_hit_active` is cleared.

  If `trigger_hit_animation()` is called while `_hit_active` is already true, the Hit clip is restarted from frame 0 (call `play("Hit", 0.0)` again). `_prior_clip` and `_prior_speed` are NOT updated in this case — they retain the value set by the original call.

- **Constraints:** One-shot; does not loop. Blend time for the Hit entry is `0.0`. Blend time for the Hit exit back to state clip uses `blend_time` export.
- **Assumptions:** The AnimationPlayer stub's `is_playing()` returns false after the clip ends. In Godot 4, a non-looping clip played to completion leaves `is_playing()` as false and `current_animation` as the name of the clip that just finished. The stub must replicate this behavior.
- **Scope:** `trigger_hit_animation()` public method and `_physics_process` Hit-active handling branch.

#### 2. Acceptance Criteria

- AC-6.1: `trigger_hit_animation()` is a public method with signature `func trigger_hit_animation() -> void`.
- AC-6.2: Calling `trigger_hit_animation()` causes `animation_player.play("Hit", 0.0)` to be called.
- AC-6.3: `_hit_active` is set to `true` after `trigger_hit_animation()` is called.
- AC-6.4: While `_hit_active` is true, the normal state-resolution dispatch (ACS-4/ACS-5) is suppressed — the controller does not call `play()` for the state clip.
- AC-6.5: When `animation_player.is_playing()` returns `false` after a Hit, `_hit_active` is set to `false` and the controller re-evaluates state and calls `play(resolved_clip, blend_time)`.
- AC-6.6: The clip transitioned to after Hit completion is determined by live state evaluation at the moment of completion, not by `_prior_clip`.
- AC-6.7: If the state is `"dead"` when Hit completes, the controller plays `"Death"` (not the prior clip).
- AC-6.8: Calling `trigger_hit_animation()` a second time while `_hit_active` is true calls `play("Hit", 0.0)` again (restart) without modifying `_prior_clip` or `_prior_speed`.

#### 3. Risk & Ambiguity Analysis

- AC-6.6 supersedes the naive "restore prior clip" approach. The prior-clip fields (`_prior_clip`, `_prior_speed`) are retained in internal state but the spec explicitly does NOT use them for resume. They may be removed from the spec if the Implementation Agent finds no use — this is flagged as a cleanup candidate.
- Risk: The completion detection relies on `is_playing()` returning false. AnimationPlayer in Godot 4 does return false after a one-shot clip ends. The stub must implement this faithfully. If the stub always returns true, Hit will never exit.
- Edge case: `trigger_hit_animation()` called when `state_machine.get_state() == "dead"` — this should still play Hit (the caller is responsible for not sending hit events to dead enemies), but the post-Hit resume will correctly switch to Death.

#### 4. Clarifying Questions

- None.

### Requirement ACS-7: Death One-Shot Animation

#### 1. Spec Summary

- **Description:** When the resolved state is `"dead"` and the Death clip is not already active, the controller calls `animation_player.play("Death", 0.0)` (zero blend time for immediate response) and sets `animation_player.speed_scale = 1.0`. After calling `play("Death", 0.0)`, the controller enters "death latched" mode tracked by `_death_latched: bool = false`. Once latched, no further `play()` calls are ever made by this controller for the lifetime of the node — all subsequent `_physics_process` ticks are no-ops for animation.

  The Death clip must hold its last frame. In Godot 4 this is achieved by not looping the clip (AnimationPlayer's loop mode for the clip is set to `ANIMATION_LOOP_NONE` — this is a responsibility of the clip authoring / import pipeline, not this script). The controller does not call `stop()` or `queue_free()` after Death.

- **Constraints:** Exactly one `play("Death", 0.0)` call for the lifetime of a given enemy. No loops. No `stop()` call after Death. `_death_latched` is irreversible once set.
- **Assumptions:** The `"Death"` clip in the imported GLB is authored as non-looping. The controller has no way to enforce loop mode at runtime; this is documented as an integration contract with `blender_animation_export`.
- **Scope:** Death-branch within `_physics_process`.

#### 2. Acceptance Criteria

- AC-7.1: When state becomes `"dead"` for the first time and `_death_latched` is false, `animation_player.play("Death", 0.0)` is called exactly once.
- AC-7.2: `_death_latched` is set to `true` after the Death clip is triggered.
- AC-7.3: On every subsequent `_physics_process` tick after `_death_latched` is true, no `play()`, `stop()`, or `speed_scale` call is made.
- AC-7.4: If `trigger_hit_animation()` is called after `_death_latched` is true, it is a no-op (Death wins; Hit is suppressed).
- AC-7.5: `animation_player.speed_scale` is set to `1.0` when Death is triggered.

#### 3. Risk & Ambiguity Analysis

- AC-7.4 is a defensive rule: callers should not send hit events to dead enemies, but the animation controller must not break if they do.
- Risk: If `_death_latched` is not set atomically before the next `_physics_process` tick, a single-frame window could trigger Death a second time. GDScript is single-threaded so this is not a real race, but the implementation must set the flag in the same call that calls `play()`.

#### 4. Clarifying Questions

- None.

### Requirement ACS-8: AnimationPlayer Stub Contract (Test Infrastructure)

#### 1. Spec Summary

- **Description:** All unit tests for `EnemyAnimationController` must be headlessly runnable via `run_tests.sh`. Because `AnimationPlayer` requires a scene tree, tests must inject a stub. The stub is an inner class or top-level class defined in the test file. The stub must faithfully implement the subset of `AnimationPlayer` that the controller uses.

  Required stub interface:
  - `func play(anim_name: String, blend_time: float = -1.0) -> void` — records the last call in `last_played_name: String` and `last_played_blend: float`; sets `current_animation = anim_name`; sets `_is_playing = true`.
  - `func stop() -> void` — sets `_is_playing = false`.
  - `func is_playing() -> bool` — returns `_is_playing`.
  - `var current_animation: String = ""` — readable/writable property.
  - `var speed_scale: float = 1.0` — readable/writable property.
  - `func simulate_clip_end() -> void` — test helper: sets `_is_playing = false` without changing `current_animation`, simulating natural one-shot completion.

  Required stub for state machine:
  - `func get_state() -> String` — returns configurable `_state: String`.
  - Stub state machine must allow `_state` to be set directly by tests between calls.

- **Constraints:** Stubs are defined in the test file, not in production code. Production code must not import or reference test stubs.
- **Assumptions:** GDScript inner classes (using `class StubAnimationPlayer:`) satisfy the duck-typing requirements since `var animation_player: Object` accepts any object assigned programmatically in tests.
- **Scope:** Test files only. No production code changes.

#### 2. Acceptance Criteria

- AC-8.1: A stub `AnimationPlayer` class exposing the five interface members above is defined in the animation controller test file.
- AC-8.2: A stub state machine class exposing `get_state() -> String` and a settable `_state` field is defined in the test file.
- AC-8.3: The controller's `animation_player` var can be assigned the stub instance without a type error at runtime.
- AC-8.4: `stub.simulate_clip_end()` sets `is_playing()` to false, enabling Hit/Death completion tests to be deterministic.
- AC-8.5: All tests pass under `run_tests.sh` (headless, no real AnimationPlayer or scene tree).

#### 3. Risk & Ambiguity Analysis

- `var animation_player: Object` (no `@export`) accepts any Object at runtime without type enforcement. Test injection via `controller.animation_player = stub_instance` works reliably in all Godot 4 versions since there is no export type constraint to enforce.
- Risk: If a future Godot version enforces typed `var` declarations at assignment, stub injection will break. This is acceptable for now; flagged for future hardening.
- Edge case: Stub `play()` must record the call even when called with only one argument (`play("Idle")`). Default parameter `blend_time = -1.0` handles this.

#### 4. Clarifying Questions

- None.
