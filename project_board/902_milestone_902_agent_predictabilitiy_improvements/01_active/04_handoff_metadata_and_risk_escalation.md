# Title

Handoff metadata schema and risk-based escalation (PASS/WARN/FAIL/ESCALATE)

# Context

Every validation gate must emit structured metadata (risk, architecture quality proxies, duplication deltas, violations/warnings) and support PASS/WARN/FAIL/ESCALATE with human review for escalations. Downstream agents consume this metadata to reason about confidence and weak areas.

# Scope

- Define versioned JSON schema for gate outputs including: `status`, `risk_score`, `architecture_score`, `test_confidence`, `duplication_delta`, `complexity_delta`, `violations`, `warnings`, `escalation_reasons`, `audit_events`.
- Implement aggregation rules when multiple tools run in one gate (ordering, dedupe, max severity).
- Define ESCALATE triggers from the MVP (governance file modifications, architecture drift heuristics, suppression abuse signals, security-sensitive paths, repeated failures) as concrete detectors where possible.
- Emit audit events in a structured append-only log under `project_board/` or `ci/artifacts/` (choose consistent, gitignored location if noisy).

# Acceptance Criteria

- JSON schema file exists and validates example outputs (test or CI validation step).
- Gate runner produces merged metadata for a multi-tool run.
- ESCALATE blocks autonomous progression in autopilot tooling OR is explicitly wired to "human required" with a clear machine-readable flag.
- WARN vs FAIL thresholds are documented and configurable without code edits (config file).

# Agent Execution Prompt

Implement handoff metadata and escalation semantics.

Goal: Add a schema + merger + severity mapping on top of the gate framework outputs. Integrate with at least the static analysis gate bundle.

Constraints:
- Keep schema backward compatible via `version` field.
- Avoid storing secrets in logs.

Expected output:
- `schema.json` (or similar) + examples + validator script.
- Documentation of thresholds in Milestone 902 README (one short section).

# Failure Handling Prompt

If blocked, ask:

- What dependency is missing? (jsonschema library, jq, etc.)
- What assumption cannot be verified? (no existing "risk score" signal)
- What ambiguity prevents completion? (what is "architecture drift" in measurable terms)

# Clarification Prompt

If unclear, ask:

- What specific ambiguity exists about where audit logs should live?
- What decision needs to be made about numeric scores vs categorical only?
- What are the possible interpretations of "test_confidence" without coverage integration?

# Dependencies

- Validation gate framework for multi-agent handoffs (orchestration, routing, remediation)
- Mandatory static analysis gate: Python, TypeScript/React, Godot, and duplication tooling

# Definition of Done

- Schema exists with examples and validation.
- At least one gate produces compliant metadata end-to-end.
